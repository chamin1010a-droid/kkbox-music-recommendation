## Chapter 1. 선형대수학 (Linear Algebra — 데이터를 숫자로 다루는 법)

> **"데이터는 행렬이고, 모델은 행렬 연산이다."**
> — 3Blue1Brown, *Essence of Linear Algebra*

### 🧭 왜 이것을 배우는가?

KKBox 대회에서 우리가 한 일을 생각해 보자:
1. **유저 정보**를 숫자로 바꿨다 (나이, 가입일, 활동량...)
2. **곡 정보**를 숫자로 바꿨다 (길이, 장르, 인기도...)
3. 이 숫자들을 **하나의 표(행렬)**로 정리했다
4. 이 표를 **LightGBM에 넣어서** 재청취 확률을 예측했다

이 과정의 **거의 모든 단계**가 선형대수다. 특히 v8에서 추가한 SVD 임베딩 — "유저와 곡의 궁합 점수"를 계산하는 것 — 은 선형대수 없이는 불가능하다. 선형대수는 데이터 과학의 **모국어**다.

### 1.1 벡터와 행렬: 데이터의 언어

#### 📌 용어 해설
- **스칼라(scalar)**: 그냥 숫자 하나. 예: `곡 길이 = 4.2분`
- **벡터(vector)**: 숫자들을 한 줄로 나열한 것. 예: `[4.2, 3, 112, 0.7]` (곡의 특성들)
- **행렬(matrix)**: 벡터들을 여러 줄로 쌓은 2차원 표. 엑셀 시트와 같다.
- **텐서(tensor)**: 3차원 이상의 배열. 사진(가로×세로×색상) 같은 것.

머신러닝에서 모든 데이터는 벡터(vector)로 표현된다. KKBox 파이프라인에서 하나의 행(row)은 "유저 × 곡" 상호작용을 나타내는 **피처 벡터(feature vector, 특성들을 모은 숫자 묶음)**다.

```
KKBox 피처 벡터 예시 (하나의 "유저-곡" 조합을 설명하는 100개+ 숫자들):

x = [user_count, song_count, artist_count, song_length_min, 
     user_svd_0, user_svd_1, ..., user_svd_31,
     song_svd_0, ..., song_svd_31,
     svd_dot, same_artist_as_prev, ...]
```

**구체적 예시 — "유저이 잔나비 '주저하는 연인들을 위해'를 다시 들을까?"**

```
이 질문에 대한 피처 벡터:
[
  user_count = 8502,        ← 유저의 총 재생 횟수
  song_count = 342,         ← 이 곡의 전체 유저 재생 횟수  
  artist_count = 5721,      ← 잔나비 전체 재생 횟수
  song_length_min = 3.8,    ← 곡 길이 3분 48초
  user_artist_ratio = 0.12, ← 유저 전체 재생 중 잔나비 비율 12%
  same_artist_as_prev = 1,  ← 직전 곡도 잔나비였음
  svd_dot = 2.34,           ← 유저-곡 궁합 점수 (높음!)
  ...
]
→ 모델 예측: target = 0.87 (87% 확률로 다시 들을 것)
```

| 개념 | 일상 비유 | 정의 | KKBox 예시 |
|------|----------|------|-----------|
| **스칼라** | 체온 하나 | 숫자 1개 | `song_length_min = 4.2` |
| **벡터** | 건강검진 결과표 한 장 | 숫자들의 1차원 나열 | 유저 SVD 임베딩 `[0.12, -0.03, ...]` |
| **행렬** | 학교 전체 성적표 | 벡터들을 쌓은 2차원 표 | 유저-곡 상호작용 행렬 |
| **텐서** | 여러 해의 성적표 묶음 | 3차원 이상 배열 | 시간별 청취 이력 |

### 1.2 내적 (Dot Product) — "궁합 점수"의 수학

> **이전 연결**: 1.1에서 데이터를 벡터로 표현하는 법을 배웠다.
> **이 절의 목적**: 두 벡터가 **얼마나 비슷한지** 측정하는 가장 기본적인 방법을 배운다.

#### 📌 용어 해설
- **내적(dot product)**: 두 벡터의 대응하는 숫자끼리 곱해서 모두 더한 값
- **코사인 유사도(cosine similarity)**: 내적을 정규화한 것. -1(정반대)~1(완전일치) 사이 값

**수식 (유니코드):**

```
u · v = u₁×v₁ + u₂×v₂ + ... + uₙ×vₙ
      = ||u|| × ||v|| × cos(θ)
```

- `||u||`: 벡터 u의 크기 (길이)
- `θ (세타)`: 두 벡터 사이의 각도
- `cos(θ)`: 각도가 0°면 1(같은 방향), 90°면 0(무관), 180°면 -1(반대)

**구체적 예시 — KKBox SVD 궁합 점수:**

```
유저의 취향 벡터 (SVD에서 자동 학습된 32개의 "잠재 취향 점수"):
  u = [0.8, 0.3, -0.5, 0.1, ...]
       ↑        ↑
       "인디록   "발라드
        좋아함"   약간 좋아함"

잔나비 '주저하는 연인들을 위해'의 특성 벡터:
  v = [0.7, 0.5, -0.2, 0.3, ...]
       ↑        ↑
       "인디록   "발라드 
        성향"    성향"

내적 계산:
  u · v = 0.8×0.7 + 0.3×0.5 + (-0.5)×(-0.2) + 0.1×0.3 + ...
        = 0.56  + 0.15   + 0.10          + 0.03     + ...
        = 2.34 (높음! → 잘 맞는 궁합)

BTS 'Butter'의 특성 벡터:
  w = [-0.3, 0.1, 0.9, 0.6, ...]
       ↑          ↑
       "인디록과    "댄스팝
        거리 멈"    성향"

  u · w = 0.8×(-0.3) + 0.3×0.1 + (-0.5)×0.9 + 0.1×0.6 + ...
        = -0.24     + 0.03    + (-0.45)     + 0.06    + ...
        = -0.42 (음수! → 취향 불일치)
```

> [!IMPORTANT]
> KKBox v8 파이프라인에서 `svd_dot` 피처가 바로 이 내적이다.
> ```python
> # pipeline_v8.py Line 368
> train['svd_dot'] = np.sum(user_vecs_train * song_vecs_train, axis=1)
> ```
> 이 한 줄이 **"이 유저와 이 곡의 궁합은 몇 점인가?"**를 계산하는 것이다.

**내적의 직관 정리:**
- 내적이 **크고 양수** → 같은 방향 → 유저와 곡의 취향이 **딱 맞음** ✅
- 내적이 **0 근처** → 직교(수직) → 유저가 이 곡에 **별 관심 없음** 😐
- 내적이 **음수** → 반대 방향 → 유저가 이 곡을 **기피할 가능성** ❌

### 1.3 행렬 분해 — "거대한 표를 작은 표 두 개로 쪼개기"

> **이전 연결**: 1.2에서 두 벡터의 궁합을 내적으로 계산하는 법을 배웠다.
> **이 절의 목적**: 수백만 유저 × 수백만 곡의 거대한 표를 효율적으로 처리하는 비밀 무기를 배운다.
> **다음 연결**: 이것이 Part 4의 추천 시스템(협업 필터링)의 수학적 기초가 된다.

#### 📌 용어 해설
- **행렬 분해(Matrix Factorization)**: 하나의 큰 행렬을 두 개의 작은 행렬의 곱으로 근사(approximation, 거의 비슷하게 표현)하는 것
- **SVD (Singular Value Decomposition, 특이값 분해)**: 행렬 분해의 가장 대표적인 방법. "보유한 정보의 중요도 순서대로 정렬한 뒤, 덜 중요한 것을 버리는 기술"
- **잠재 벡터(latent vector)**: "눈에 보이지 않는 숨겨진 특성"을 숫자화한 것. 예: "이 유저가 인디록을 좋아하는 정도"는 데이터에 직접 없지만, SVD가 자동으로 발견한다
- **차원(dimension)**: 벡터가 가진 숫자의 개수. 32차원 = 32개의 숫자로 표현

#### 핵심 아이디어

```
원본: 유저-곡 상호작용 행렬 R (300만 유저 × 500만 곡)
  → 99.99%가 비어있음 (대부분의 유저는 대부분의 곡을 안 들었으니까)
  → 직접 다루기엔 너무 거대함

분해: R ≈ U × Σ × Vᵀ  (SVD의 완전한 형태)

  U: 유저 행렬 (300만 × k) — 각 유저를 k개의 "숨겨진 취향 점수"로 표현
  Σ: 특이값 행렬 (k × k) — 각 차원의 "중요도" (대각선에 값, 나머지는 0)
  V: 곡 행렬 (500만 × k) — 각 곡을 k개의 "숨겨진 특성 점수"로 표현
  
  여기서 k = 32 (KKBox에서 선택한 값)
```

> [!NOTE]
> **SVD의 핵심은 "중요한 것만 남기는 것"이다.**
> 
> R = U × Σ × Vᵀ 에서 Σ(시그마)는 각 차원의 중요도를 나타낸다.
> 중요도가 높은 상위 32개 차원만 남기고 나머지를 **버린다.**
> 
> 이것은 단순히 곱을 쪼개는 게 아니라, **핵심 정보는 보존하면서 노이즈(잡음)를 제거하는 과정**이다.
> 음악 녹음에서 주요 주파수만 남기고 배경 잡음을 제거하는 것과 비슷하다.

**구체적 비유 — "음악 취향 설문조사":**

```
원래 방식 (원본 행렬):
  "유저은 50만 곡 각각에 대해 들었는지 안 들었는지를 0/1로 기록"
  → 50만 개의 숫자가 필요
  → 99.99%가 0 (안 들어본 곡)
  → 비효율적이고, "비슷한 곡"이라는 관계도 안 보임

SVD 방식 (분해 후):
  "유저의 취향을 32개의 숨겨진 차원으로 압축"
  차원 1: "인디록 선호도" = 0.8   (높음)
  차원 2: "발라드 선호도" = 0.3   (보통)
  차원 3: "EDM 선호도" = -0.5    (낮음)
  차원 4: "한국어 선호도" = 0.7   (높음)
  ...
  차원 32: "소소한 무언가" = 0.01  (거의 의미 없음)
  
  → 32개의 숫자만으로 50만 곡에 대한 취향을 압축!
  → 게다가 이 차원들은 SVD가 데이터에서 **자동으로 발견**한 것
```

**KKBox에서의 실제 코드:**

```python
# pipeline_v8.py — SVD 임베딩
svd = TruncatedSVD(n_components=32, random_state=42, n_iter=10)
user_factors = svd.fit_transform(interaction)   # 유저 행렬 U: (300만, 32)
song_factors = svd.components_.T                # 곡 행렬 V: (500만, 32)

# 궁합 점수 = 유저 벡터와 곡 벡터의 내적 (1.2에서 배운 것!)
train['svd_dot'] = np.sum(user_vecs * song_vecs, axis=1)
```

**왜 32차원인가? — `explained_variance_ratio_` (설명된 분산 비율)**

```
특이값의 크기 (= 각 차원의 중요도):
  차원 1:  ████████████████████████  (23.1%) ← 가장 핵심적인 취향 축
  차원 2:  ██████████████████       (15.7%)
  차원 3:  ████████████             (10.2%)
  ...
  차원 32: ██                       (0.3%)  ← 거의 노이즈
  ─────────────────────────────────
  상위 32개 합계: 약 85% 설명
  
  → 원래 50만 차원의 정보 중 85%를 단 32개로 보존!
  → 나머지 15%는 대부분 노이즈였으므로, 오히려 제거하는 게 이득
```

> [!TIP]
> 3Blue1Brown의 "선형변환" 관점으로 이해하면:
> SVD는 고차원 공간의 데이터를 **정보가 가장 많은 방향(축)**으로 정렬한 뒤,
> 상위 k개 방향만 남기는 것이다. 나머지 방향은 정보보다 노이즈가 많으므로 버린다.

#### 🔢 수학 예시 — 작은 행렬로 SVD 전 과정 따라하기

4명의 유저가 5곡을 들었는지(1) 안 들었는지(0)를 기록한 행렬 R:

```
          곡A  곡B  곡C  곡D  곡E
유저1  [  1    1    0    0    0  ]
유저2  [  1    1    1    0    0  ]
유저3  [  0    0    1    1    1  ]
유저4  [  0    0    0    1    1  ]
```

눈으로 봐도 패턴이 보인다:
- 유저1,2는 곡A,B를 좋아함 → "록 취향"
- 유저3,4는 곡D,E를 좋아함 → "발라드 취향"
- 곡C는 양쪽 다 약간 걸침 → "크로스오버"

**Step 1: SVD 분해 — R = U × Σ × Vᵀ**

SVD를 수행하면 세 행렬로 분해된다 (계산은 컴퓨터가 수행):

```
U (4×4, 유저 행렬):           Σ (4×4, 특이값 대각행렬):      Vᵀ (4×5, 곡 행렬의 전치)
                              대각선 값 = 특이값 (중요도)

[ -0.44  -0.53   ...  ... ]   [ 2.16  0     0     0   ]   [ -0.31  -0.31  -0.41  -0.53  -0.53 ]
[ -0.63  -0.30   ...  ... ]   [ 0     1.83  0     0   ]   [ -0.56  -0.56   0.17   0.42   0.42 ]
[ -0.53   0.60   ...  ... ]   [ 0     0     0.68  0   ]   [  ...    ...   ...    ...    ...   ]
[ -0.33   0.51   ...  ... ]   [ 0     0     0     0.12]   [  ...    ...   ...    ...    ...   ]
```

특이값 Σ를 보자:
```
  σ₁ = 2.16  ← 가장 중요한 차원 ("록 vs 발라드" 축)
  σ₂ = 1.83  ← 두 번째 중요    ("활발 vs 비활발" 축)
  σ₃ = 0.68  ← 별로 안 중요    (곡C의 크로스오버 잡음)
  σ₄ = 0.12  ← 거의 노이즈     (무의미한 변동)
```

**Step 2: 절삭(Truncation) — 상위 k=2개만 남기기**

핵심: σ₃, σ₄를 **버린다**. 이것이 "차원 축소"다.

```
원래: R = U(4×4) × Σ(4×4) × Vᵀ(4×5)  →  정확한 복원
절삭: R ≈ U₂(4×2) × Σ₂(2×2) × V₂ᵀ(2×5) →  근사 복원 (핵심만 보존)

U₂ (4×2):             Σ₂ (2×2):           V₂ᵀ (2×5):
[ -0.44  -0.53 ]      [ 2.16  0    ]      [ -0.31  -0.31  -0.41  -0.53  -0.53 ]
[ -0.63  -0.30 ]      [ 0     1.83 ]      [ -0.56  -0.56   0.17   0.42   0.42 ]
[ -0.53   0.60 ]
[ -0.33   0.51 ]
```

**Step 3: 근사 복원 — R̂ = U₂ × Σ₂ × V₂ᵀ 계산**

먼저 U₂ × Σ₂ (유저 임베딩에 중요도를 반영):

```
U₂ × Σ₂:
  유저1: [-0.44×2.16, -0.53×1.83] = [-0.95, -0.97]
  유저2: [-0.63×2.16, -0.30×1.83] = [-1.36, -0.55]
  유저3: [-0.53×2.16,  0.60×1.83] = [-1.14,  1.10]
  유저4: [-0.33×2.16,  0.51×1.83] = [-0.71,  0.93]
```

이제 이것과 V₂ᵀ를 곱하면 복원된 행렬 R̂:

```
R̂ (복원 결과, 반올림):
           곡A   곡B   곡C   곡D   곡E
유저1  [  0.84  0.84  0.23 -0.08 -0.08 ]  ← 원본은 [1,1,0,0,0] → 거의 맞음!
유저2  [  0.73  0.73  0.46  0.49  0.49 ]  ← 원본은 [1,1,1,0,0] → 곡D,E에 약간 오차
유저3  [  0.26  0.26  0.66  0.97  0.97 ]  ← 원본은 [0,0,1,1,1] → 거의 맞음!
유저4  [  0.30  0.30  0.45  0.77  0.77 ]  ← 원본은 [0,0,0,1,1] → 곡C에 약간 오차
```

**Step 4: 해석 — 이 숫자들이 의미하는 것**

```
2차원으로 압축된 유저 임베딩:
  유저1 = [-0.95, -0.97]  ↘ 록 취향, 가벼운 청취자
  유저2 = [-1.36, -0.55]  ↙ 록 취향, 활발한 청취자 
  유저3 = [-1.14,  1.10]  ↗ 발라드 취향, 활발한 청취자
  유저4 = [-0.71,  0.93]  ↗ 발라드 취향, 가벼운 청취자

2차원으로 압축된 곡 임베딩:
  곡A = [-0.31, -0.56]  ← 록 영역
  곡B = [-0.31, -0.56]  ← 록 영역 (곡A와 비슷 → 비슷한 곡!)
  곡C = [-0.41,  0.17]  ← 중간 (크로스오버)
  곡D = [-0.53,  0.42]  ← 발라드 영역
  곡E = [-0.53,  0.42]  ← 발라드 영역 (곡D와 비슷)

궁합 점수 (내적):
  유저1 · 곡A = (-0.95)×(-0.31) + (-0.97)×(-0.56) = 0.29 + 0.54 = 0.84 ← 높음! ✅
  유저1 · 곡D = (-0.95)×(-0.53) + (-0.97)×(0.42) = 0.50 - 0.41 = 0.09 ← 낮음  ❌
  유저3 · 곡D = (-1.14)×(-0.53) + (1.10)×(0.42)  = 0.60 + 0.46 = 1.06 ← 높음! ✅
```

> [!IMPORTANT]
> **이것이 SVD의 전체 과정이다:**
> 1. 큰 행렬 R을 U × Σ × Vᵀ로 **분해**
> 2. Σ에서 작은 특이값(노이즈)을 **제거** (k개만 남김)
> 3. 남은 U₂와 V₂가 각각 **유저 임베딩**, **곡 임베딩**
> 4. 유저·곡 내적 = **궁합 점수** (KKBox의 `svd_dot`)
>
> 4×5 행렬에서 2차원으로 줄였는데도 원본의 핵심 패턴(록/발라드 구분)이 보존되었다.
> 실제 KKBox에서는 300만×500만 행렬을 32차원으로 줄이는 것이다.

#### 🔢 행렬 곱 복습 — SVD 계산을 따라하기 위한 기초

행렬 곱이 어떻게 작동하는지 모르면 위 SVD 계산을 따라갈 수 없다. 기본부터:

```
A (2×3):         B (3×2):         A × B (2×2):
[ 1  2  3 ]      [ 7  8 ]         [ 1×7+2×9+3×11   1×8+2×10+3×12 ]
[ 4  5  6 ]      [ 9  10]         [ 4×7+5×9+6×11   4×8+5×10+6×12 ]
                 [ 11 12]         
                                  = [ 7+18+33    8+20+36  ]   = [ 58   64 ]
                                    [ 28+45+66   32+50+72 ]     [139  154 ]

규칙: C[i][j] = A의 i번째 행과 B의 j번째 열의 내적

  C[1][1] = [1,2,3] · [7,9,11] = 1×7 + 2×9 + 3×11 = 58
  C[1][2] = [1,2,3] · [8,10,12] = 1×8 + 2×10 + 3×12 = 64
  ...

핵심: A의 열 수(3) = B의 행 수(3)가 같아야 곱 가능
     결과 크기 = A의 행 수(2) × B의 열 수(2)
```

### 1.4 고유값 분해 — PCA의 기초

> **이전 연결**: 1.3에서 SVD로 행렬을 분해하는 법을 배웠다.
> **이 절의 목적**: SVD의 이론적 배경인 고유값 분해와, 자주 쓰이는 PCA를 이해한다.
> **실무 요약**: PCA와 SVD는 목적이 같다 — "핵심 정보만 남기고 차원을 줄이는 것".

#### 📌 용어 해설
- **고유값(eigenvalue)**: 행렬이 특정 방향으로 작용할 때, 그 방향의 늘어남/줄어듦 정도
- **고유벡터(eigenvector)**: 행렬에 의해 **방향이 변하지 않는** 특별한 벡터
- **PCA (Principal Component Analysis, 주성분 분석)**: 데이터의 분산이 가장 큰 방향을 찾아 그 방향으로 차원을 줄이는 기법 (SVD와 매우 유사)

**고유값 분해 핵심:**

```
정방 행렬 A에 대해:
  A × v = λ × v

  v: 고유벡터 — 행렬 A로 변환해도 "방향이 안 바뀌는" 특별한 벡터
  λ: 고유값  — 그 방향으로 얼마나 늘어나는지 (스케일 변화량)
```

**비유:**

```
지진이 발생하면 건물이 흔들린다.
대부분의 점은 여러 방향으로 복잡하게 움직이지만,
특정 축(고유벡터)을 따르는 점은 그 축 위에서만 앞뒤로 움직인다.
고유값은 그 축에서 "얼마나 크게 흔들리는지"다.

데이터도 마찬가지. 고유벡터 = "데이터가 가장 크게 변하는 방향"
                    고유값 = "그 방향으로의 변동 크기"
```

**PCA vs SVD — 언제 뭘 쓰나:**

| | PCA | TruncatedSVD |
|---|---|---|
| **적용 대상** | 밀집(dense) 데이터 | 희소(sparse) 행렬 |
| **전처리** | 평균을 빼야 함 (중심화) | 필요 없음 |
| **KKBox 적합성** | ❌ (유저-곡 행렬이 99.9% 빈칸) | ✅ 사용함 |
| **원리** | 공분산 행렬의 고유값 분해 | 원본 행렬의 특이값 분해 |

> [!NOTE]
> **결론**: PCA와 SVD는 **같은 목적의 다른 구현**이다.
> 둘 다 "중요한 차원 k개만 남기고 나머지를 버린다"는 핵심은 동일.
> KKBox처럼 데이터가 대부분 비어있는(sparse) 경우에는 TruncatedSVD가 적합하다.

#### 🔢 수학 예시 — 2×2 행렬의 고유값 분해 직접 계산

```
행렬 A:
  [ 4  2 ]
  [ 1  3 ]

"A × v = λ × v" 를 만족하는 λ(고유값)와 v(고유벡터)를 찾자.
```

**Step 1: 특성방정식(characteristic equation) 세우기**

```
det(A - λI) = 0   (I는 단위행렬)

A - λI = [ 4-λ   2  ]
         [  1   3-λ ]

행렬식(determinant):
  det = (4-λ)(3-λ) - 2×1
      = 12 - 4λ - 3λ + λ² - 2
      = λ² - 7λ + 10
      = (λ - 5)(λ - 2) = 0

→ λ₁ = 5,  λ₂ = 2   ← 이것이 고유값!
```

**Step 2: 각 고유값에 대응하는 고유벡터 구하기**

```
λ₁ = 5일 때: (A - 5I)v = 0
  [ -1   2 ] [ v₁ ]   [ 0 ]
  [  1  -2 ] [ v₂ ] = [ 0 ]

  -v₁ + 2v₂ = 0  →  v₁ = 2v₂
  v₂ = 1로 놓으면 → v₁ = [2, 1]ᵀ  (정규화: [2/√5, 1/√5])

λ₂ = 2일 때: (A - 2I)v = 0
  [ 2   2 ] [ v₁ ]   [ 0 ]
  [ 1   1 ] [ v₂ ] = [ 0 ]

  2v₁ + 2v₂ = 0  →  v₁ = -v₂
  v₂ = 1로 놓으면 → v₂ = [-1, 1]ᵀ  (정규화: [-1/√2, 1/√2])
```

**Step 3: 검증 — A × v = λ × v 가 정말 성립하는가?**

```
A × v₁ = [ 4  2 ] [ 2 ]   [ 4×2 + 2×1 ]   [ 10 ]       [ 2 ]
         [ 1  3 ] [ 1 ] = [ 1×2 + 3×1 ] = [  5 ] = 5 × [ 1 ]  ✅

→ 행렬 A를 벡터 [2,1]에 적용하면, 방향은 그대로이고 크기만 5배가 된다!
   이것이 "고유벡터는 방향이 안 바뀐다"의 의미.

A × v₂ = [ 4  2 ] [-1 ]   [ -4+2 ]   [ -2 ]       [ -1 ]
         [ 1  3 ] [ 1 ] = [ -1+3 ] = [  2 ] = 2 × [  1 ]  ✅
```

**Step 4: PCA에서의 의미**

```
만약 A가 데이터의 공분산 행렬이라면:

  고유값 λ₁ = 5 → 첫 번째 주성분 방향 v₁ = [2,1]에서 분산이 5
  고유값 λ₂ = 2 → 두 번째 주성분 방향 v₂ = [-1,1]에서 분산이 2

  총 분산 = 5 + 2 = 7
  첫 번째 주성분의 설명력 = 5/7 = 71.4%
  두 번째 주성분의 설명력 = 2/7 = 28.6%

  만약 1차원으로 줄인다면? → v₁ 방향만 남기고 v₂ 방향을 버림
  → 정보의 71.4%를 보존하면서 차원을 절반으로 줄인다!
```

### 1.5 미적분학: "최적의 답을 찾는 법"

> **이전 연결**: 1.1~1.4에서 데이터를 숫자로 표현하고 줄이는 법을 배웠다.
> **이 절의 목적**: 모델의 정답(파라미터)을 찾는 핵심 도구 — 경사하강법을 배운다.
> **다음 연결**: Part 2~4의 **모든 모델**은 경사하강법의 변형으로 학습한다.

#### 📌 용어 해설
- **파라미터(parameter, 매개변수)**: 모델이 데이터에서 **학습하는 값**. "정답을 맞추기 위해 조절하는 손잡이"
  - KKBox LightGBM의 경우: 각 트리의 분할 지점과 리프 값
  - 선형회귀의 경우: 기울기(slope)와 절편(intercept)
- **하이퍼파라미터(hyperparameter)**: 사람이 **미리 정하는 설정값**. 모델이 학습하지 않음.
  - 예: `learning_rate = 0.1`, `n_estimators = 2000`, `num_leaves = 255`
- **그래디언트(gradient, 기울기/경사)**: "이 방향으로 가면 답에서 멀어진다/가까워진다"를 알려주는 나침반
- **손실함수(loss function)**: "현재 모델의 예측이 정답과 얼마나 다른가"를 측정하는 점수. **낮을수록 좋다**

**편미분 — "한 번에 하나의 손잡이만 돌리기"**

```
모델에 손잡이(파라미터)가 100개 있다고 하자.
편미분 = "다른 99개는 고정하고, 손잡이 1번만 살짝 돌렸을 때 결과가 얼마나 변하는가?"

예: 3개의 손잡이를 가진 간단한 모델
  f(x₁, x₂, x₃) = x₁² + 3x₂ + x₁x₃

  ∂f/∂x₁ = 2x₁ + x₃     ← x₁을 살짝 움직이면 f가 이만큼 변한다
  ∂f/∂x₂ = 3              ← x₂를 살짝 움직이면 f가 3만큼 변한다 (항상 일정)
  ∂f/∂x₃ = x₁             ← x₃을 살짝 움직이면 f가 x₁만큼 변한다
```

**그래디언트 — "모든 손잡이의 나침반을 한 번에 보기"**

```
그래디언트 = 모든 편미분을 벡터로 묶은 것
∇f = [∂f/∂x₁, ∂f/∂x₂, ..., ∂f/∂xₙ]

의미: "함수 f가 가장 빠르게 증가하는 방향"
→ 우리는 손실(Loss)를 줄이고 싶으니까, 그래디언트의 반대 방향으로 가면 된다!
```

**경사하강법 (Gradient Descent) — "산에서 내려오기"**

```
목표: 손실함수 L을 최소화하는 파라미터 θ를 찾자

알고리즘:
  1. 랜덤한 위치에서 시작 (초기 파라미터)
  2. 현재 위치에서 "어느 방향이 내리막인지" 확인 (그래디언트 계산)
  3. 그 방향으로 조금 이동 (파라미터 업데이트)
  4. 더 이상 내려갈 곳이 없을 때까지 반복

  θ_새로운 = θ_현재 - η × ∇L(θ_현재)

  η (에타, "학습률"): 한 번에 얼마나 이동할지 결정하는 보폭 크기
```

**구체적 비유 — "안개 속에서 산 내려오기":**

```
상황: 당신은 깜깜한 밤, 안개 속의 산꼭대기에 있다. 
      가장 낮은 골짜기(= 최소 손실)로 내려가야 한다.

전략:
  1. 발로 주변 지면의 기울기를 느낀다 (= 그래디언트 계산)
  2. 가장 가파르게 내려가는 방향으로 한 발 내딛는다 (= 파라미터 업데이트)
  3. 반복

학습률(η)의 영향:
  η = 큰 값: 한 번에 큰 보폭 → 빨리 내려가지만 골짜기를 지나칠 수 있음 (발산)
  η = 작은 값: 조심스러운 보폭 → 안전하지만 시간이 너무 오래 걸림
  η = 적당한 값: 빠르고 안정적 (KKBox에서 0.1을 사용)
```

> [!TIP]
> KKBox v7→v8에서 learning_rate를 0.1로 고정한 이유:
> "트리를 늘려도(보폭 수를 늘려도) 로컬 AUC만 상승하고 Kaggle은 0.65 고정"
> → 문제는 보폭(학습률)이 아니라 **산의 모양 자체(피처 품질)**였다.
> → 그래서 SVD 임베딩과 시퀀스 피처를 추가하여 "산의 구조"를 바꾼 것이다.

#### 🔢 수학 예시 — 경사하강법 4단계 직접 계산

가장 간단한 문제: 손실함수 L(w) = (w - 3)² 을 최소화하는 w를 찾자.
(정답은 당연히 w = 3이지만, 경사하강법이 어떻게 찾아가는지 과정을 보자)

```
설정:
  손실함수: L(w) = (w - 3)²
  그래디언트: dL/dw = 2(w - 3)   ← 미분으로 구함
  학습률: η = 0.3
  초기값: w₀ = 0 (랜덤 시작)

─── 반복 1 ───
  현재 w₀ = 0
  그래디언트: dL/dw = 2×(0-3) = -6    ← "오른쪽으로 가면 손실이 줄어든다"
  업데이트: w₁ = 0 - 0.3×(-6) = 0 + 1.8 = 1.8
  손실: L(1.8) = (1.8-3)² = 1.44     ← 9에서 1.44로 감소! ✅

─── 반복 2 ───
  현재 w₁ = 1.8
  그래디언트: dL/dw = 2×(1.8-3) = -2.4
  업데이트: w₂ = 1.8 - 0.3×(-2.4) = 1.8 + 0.72 = 2.52
  손실: L(2.52) = (2.52-3)² = 0.23   ← 1.44에서 0.23으로 감소! ✅

─── 반복 3 ───
  현재 w₂ = 2.52
  그래디언트: dL/dw = 2×(2.52-3) = -0.96
  업데이트: w₃ = 2.52 - 0.3×(-0.96) = 2.52 + 0.288 = 2.808
  손실: L(2.808) = (2.808-3)² = 0.037 ← 또 감소! ✅

─── 반복 4 ───
  현재 w₃ = 2.808
  그래디언트: dL/dw = 2×(2.808-3) = -0.384
  업데이트: w₄ = 2.808 - 0.3×(-0.384) = 2.808 + 0.115 = 2.923
  손실: L(2.923) = (2.923-3)² = 0.006 ← 거의 0! ✅
```

```
수렴 과정 시각화:
  
  w:     0 ──→ 1.8 ──→ 2.52 ──→ 2.81 ──→ 2.92 ──→ ... ──→ 3.0 (정답!)
  Loss:  9     1.44    0.23     0.037    0.006   ...      0
  
  그래디언트가 점점 작아지면서 → 보폭도 점점 작아지면서 → 정답에 수렴
```

**학습률(η)이 다르면?**

```
η = 0.1 (너무 작음):
  w₀=0 → w₁=0.6 → w₂=1.08 → w₃=1.46 → ... 30번 반복해도 w=2.8 (느림!)

η = 0.3 (적당):
  w₀=0 → w₁=1.8 → w₂=2.52 → w₃=2.81 → ... 10번이면 w≈3.0 ✅

η = 1.1 (너무 큼):
  w₀=0 → w₁=6.6 → w₂=-0.72 → w₃=8.17 → ... 발산! ❌
  (정답 3을 지나쳐서 왔다갔다 하면서 점점 멀어짐)
```

**2차원 예시 — 파라미터가 2개일 때:**

```
손실함수: L(w₁, w₂) = (w₁ - 1)² + (w₂ + 2)²
정답: w₁ = 1, w₂ = -2

그래디언트: ∇L = [2(w₁-1), 2(w₂+2)]
학습률: η = 0.3
초기값: w₁ = 0, w₂ = 0

반복 1:
  ∇L = [2(0-1), 2(0+2)] = [-2, 4]
  w₁ = 0 - 0.3×(-2) = 0.6
  w₂ = 0 - 0.3×(4)  = -1.2

반복 2:
  ∇L = [2(0.6-1), 2(-1.2+2)] = [-0.8, 1.6]
  w₁ = 0.6 - 0.3×(-0.8) = 0.84
  w₂ = -1.2 - 0.3×(1.6) = -1.68

반복 3:
  ∇L = [2(0.84-1), 2(-1.68+2)] = [-0.32, 0.64]
  w₁ = 0.84 - 0.3×(-0.32) = 0.936
  w₂ = -1.68 - 0.3×(0.64) = -1.872

→ (w₁, w₂) = (0, 0) → (0.6, -1.2) → (0.84, -1.68) → (0.94, -1.87) → ... → (1, -2)
   두 파라미터가 동시에 각자의 정답을 향해 수렴!
```

> 실제 딥러닝에서는 파라미터가 **수백만~수십억 개**이고 손실함수가 훨씬 복잡하지만,
> 원리는 위 예시와 **정확히 동일**하다: "그래디언트 계산 → 반대 방향으로 이동 → 반복".

## Chapter 2. 확률과 통계의 기초

> **"확률은 불확실성의 언어이고, 통계는 데이터로 불확실성을 줄이는 과학이다."**
> — MIT 18.650 강의 서두

### 🧭 왜 이것을 배우는가?

KKBox에서 "이 유저가 이 곡을 다시 들을 확률이 87%"라고 예측하는데, 이 **"확률"**이 정확히 뭘 의미하는가? 그리고 그 예측을 **"신뢰할 수 있는가"**를 어떻게 판단하는가?

음악 프로젝트에서 "스킵률이 증가하면 재생량이 감소한다"는 가설을 세웠을 때, 이것이 **우연인지 진짜 패턴인지** 어떻게 구분하는가?

이 질문들에 답하는 도구가 **확률과 통계**다.

---

### 2.1 확률의 기본 — 가장 직관적인 것부터

#### 📌 용어 해설
- **확률(probability)**: 어떤 일이 일어날 가능성. 0(절대 안 일어남)~1(반드시 일어남)
- **조건부 확률(conditional probability)**: "B가 이미 일어났을 때, A가 일어날 확률"
- **베이즈 정리(Bayes' theorem)**: "원인→결과"의 확률을 알 때, "결과→원인"의 확률을 역추적하는 공식

**확률의 기본 공리:**

```
1. 확률은 항상 0 이상: P(A) ≥ 0
2. 전체 가능성의 합 = 1: P(Ω) = 1
3. 동시에 일어날 수 없는 사건들의 확률 = 각각의 합:
   P(A 또는 B) = P(A) + P(B)  (A, B가 동시에 일어날 수 없을 때)
```

**베이즈 정리:**

```
              P(B|A) × P(A)
  P(A|B) = ─────────────────
                 P(B)
```

**구체적 예시 — KKBox Target Encoding의 직관:**

전체 데이터 1000건이 있다고 하자. 분할표로 정리하면:

```
                     B(아티스트 5회+)    B̅(5회 미만)     합계
A (재청취 O, y=1)         150              350          500
A̅ (재청취 X, y=0)          50              450          500
합계                       200              800         1000
```

이 표에서 각 확률을 직접 셀 수 있다:

```
P(A) = 500/1000 = 0.50     전체 중 재청취 비율
P(B) = 200/1000 = 0.20     전체 중 아티스트 5회+ 비율
P(B|A) = 150/500 = 0.30    재청취한 곡 중에서 아티스트 5회+인 비율
```

**우리가 알고 싶은 것: P(A|B) = "아티스트 5회+ 들은 경우, 재청취 확률은?"**

**방법 1 — 직접 세기 (가장 직관적):**

```
아티스트 5회+ 들은 200건 중 재청취한 건 = 150건
P(A|B) = 150/200 = 0.75
```

**방법 2 — 베이즈 정리:**

```
P(A|B) = P(B|A) × P(A) / P(B)
       = 0.30 × 0.50 / 0.20
       = 0.75   ← 직접 센 것과 동일! ✅
```

```
해석:
  전체 재청취율 = 50%인데,
  "이 아티스트를 5번 이상 들은" 경우로 좁히면 → 재청취율이 75%로 올라간다!
  
  Target Encoding = 이 P(A|B)를 계산해서 피처로 쓰는 것
  "잔나비"의 TE 값 = "잔나비를 많이 들은 유저들의 평균 재청취율" = 0.75
```

---

### 2.2 주요 확률분포 — "데이터의 모양"

> **왜 분포를 배우는가?**
> "이 데이터가 어떤 모양(분포)을 따르는지"를 알면, 적절한 모델과 분석 방법을 선택할 수 있다.

#### 이산 분포 (결과가 셀 수 있는 경우)

| 분포 | 의미 | KKBox/음악 예시 |
|------|------|---------------|
| **베르누이(Bernoulli)** | 성공/실패 두 가지 중 하나 | 이 곡을 다시 듣는다(1) vs 안 듣는다(0) = KKBox target |
| **이항(Binomial)** | n번 시도 중 성공 횟수 | 10곡 추천 중 실제 재청취한 곡 수 |
| **포아송(Poisson)** | 단위 시간/공간 당 발생 횟수 | 1시간 동안 유저가 곡을 재생한 횟수 |

**구체적 예시 — 베르누이 분포와 KKBox:**

```
KKBox의 target 컬럼은 베르누이 분포를 따른다:
  0 (다시 안 듣는다): 50% 확률
  1 (다시 듣는다):    50% 확률

우리 모델이 하는 일:
  "아, 이 유저-곡 조합은 p = 0.87이구나" (87% 확률로 1)
  → 피처 벡터를 받아서, 이 p 값을 예측하는 것이 LightGBM의 역할
```

#### 연속 분포 (결과가 연속적인 경우)

| 분포 | 모양 | KKBox/음악 예시 |
|------|------|---------------|
| **정규분포(Normal)** | 종 모양 (가운데 높고 양쪽 낮음) | SVD 임베딩 값의 분포 |
| **지수분포(Exponential)** | 왼쪽 높고 오른쪽으로 급감 | 곡 간 재생 간격 (time_gap_seconds) |

**구체적 예시 — 지수분포와 "곡 간 재생 간격":**

```
먼저 지수분포의 모양:
  
  확률
  높음 |█
       |██
       |████
       |████████
       |█████████████
  낮음 |███████████████████████████████
       └──────────────────────────────── 시간 (분)
       0    3    10     30    60

특징: "짧은 간격이 가장 흔하고, 긴 간격은 드물다"
```

```
음악 프로젝트에서의 적용:

  "곡 간 재생 간격" = 이전 곡 끝난 시점 ~ 다음 곡 시작한 시점

  보통 유저 행동:
    곡 끝남 → 0~5초 후 다음 곡 재생 (자동 재생 or 바로 선택)  ← 가장 흔함
    곡 끝남 → 1~5분 후 다음 곡 재생 (잠깐 메뉴 탐색)        ← 가끔
    곡 끝남 → 30분+ 후 다음 곡 재생 (앱 종료했다 복귀)       ← 드묾

  이 패턴이 정확히 지수분포의 모양이다:
    짧은 간격 = 확률 높음
    긴 간격 = 확률 낮음

  만약 "곡 간 간격이 정확히 3분 48초"인 기록이 연속 10번 나온다면?
    → 곡 길이가 3분 48초인 곡을 반복 재생하고 있을 가능성이 높다
    → 지수분포에서 "동일한 간격이 정확히 반복되는 것"은 극히 드물기 때문
    → 이것이 음악 프로젝트에서 "반복재생 여부 역산"에 활용한 아이디어
```

---

### 2.3 기대값, 분산, 공분산 — "데이터를 요약하는 세 가지 숫자"

#### 📌 용어 해설
- **기대값(expected value, 평균)**: "앞으로 이 실험을 무한히 반복하면, 평균적으로 이 값이 나올 것이다"
- **분산(variance)**: "데이터가 평균에서 얼마나 흩어져 있는가" (크면 = 불안정, 작으면 = 안정)
- **공분산(covariance)**: "두 변수가 함께 움직이는 정도" (양수 = 같이 증가, 음수 = 하나 증가 → 다른 하나 감소)

```
기대값 (평균):
  동전을 무한히 던지면, 앞면 나올 확률의 기대값 = 0.5
  KKBox 전체 target의 기대값 = 약 0.50 (50%가 재청취)

분산:
  "아티스트A" 재생 횟수: [100, 105, 98, 103, 99] → 분산 작음 (안정적)
  "아티스트B" 재생 횟수: [2, 500, 0, 150, 3]  → 분산 큼 (불안정)
  → 분산이 큰 아티스트 = 히트곡 하나에 의존 (Tier A)
  → 분산이 작은 아티스트 = 꾸준히 들리는 곡이 여러 개 (Tier S)

공분산:
  X = 전반부 스킵률,  Y = 후반부 재생량 변화 (log2 스케일)
  Cov(X, Y) < 0 → 스킵이 늘면 재생량이 줄어든다 (음의 공분산)
  → 이것이 regression_analysis.py에서 검증한 핵심 가설!
```

---

### 2.4 추정 이론 — "데이터로 모집단을 추측하기"

> **이전 연결**: 2.1~2.3에서 확률의 기본 개념을 배웠다.
> **이 절의 목적**: "일부 데이터(표본)만 보고 전체(모집단)의 특성을 추측"하는 방법을 배운다.
> **KKBox 연결**: 우리 모델이 학습 데이터에서 배운 패턴이 **테스트 데이터에서도 통하는가?**

#### 📌 용어 해설
- **모집단(population)**: 관심 대상 전체. 예: KKBox의 모든 유저-곡 상호작용 (과거+미래)
- **표본(sample)**: 실제로 관찰한 일부. 예: KKBox train.csv의 730만 행
- **추정량(estimator)**: 표본으로 모집단을 추측하는 공식. 예: "표본 평균 = 모평균의 추정량"
- **MLE (Maximum Likelihood Estimation, 최대우도추정)**: "이 데이터가 나올 가능성을 최대화하는 파라미터를 찾는 것"

**MLE의 직관:**

```
질문: "KKBox 유저들의 전체 재청취율(p)은 얼마인가?"

MLE의 사고방식:
  만약 p = 0.3이라면? → 우리 데이터(50%가 재청취)가 나올 확률이 매우 낮음
  만약 p = 0.5라면? → 우리 데이터가 나올 확률이 최대
  만약 p = 0.7이라면? → 우리 데이터가 나올 확률이 또 낮아짐
  
  → p = 0.5가 MLE 추정값 (이 데이터를 "가장 잘 설명하는" 값)
```

**LightGBM과 MLE의 관계:**

```
LightGBM이 이진 분류를 학습할 때 최적화하는 목적함수:

  L = -(1/n) × Σ [yᵢ × log(pᵢ) + (1-yᵢ) × log(1-pᵢ)]

이것이 Binary Cross-Entropy(= 이진 교차 엔트로피)이며,
수학적으로 "베르누이 분포의 음의 로그 우도"와 동일하다.

쉽게 말해:
  "재청취한 곡(y=1)에 높은 확률(p)을, 재청취 안 한 곡(y=0)에 낮은 확률(p)을 
   부여하도록 모델을 학습시킨다"
```

**추정량이 좋은지 나쁜지 판단하는 기준:**

| 성질 | 의미 | 일상 비유 |
|------|------|----------|
| **불편향성** (Unbiased) | 평균적으로 정답을 맞춤 | 과녁 중앙에 모여있는 화살 (편향 없음) |
| **일치성** (Consistent) | 데이터가 많을수록 정확해짐 | 연습할수록 실력이 느는 선수 |
| **효율성** (Efficient) | 같은 양의 데이터로 가장 정확 | 같은 시간 공부해도 가장 성적이 좋은 학생 |

> [!WARNING]
> **KKBox에서의 추정 실패 사례 — 데이터 누수(Data Leakage):**
> 
> v4 파이프라인에서 Target Encoding을 OOF 없이 적용했을 때:
> - 로컬 AUC: 0.96 (겉보기에 완벽한 추정)
> - Kaggle AUC: 0.58 (실제로는 랜덤 추측보다 약간 나은 수준)
> 
> 원인: 학습 데이터의 **정답(target)**이 피처에 직접 섞여 들어감
> → 모델이 "정답을 외운 것"이지 "패턴을 학습한 것"이 아니었다
> → 편향은 낮아 보이지만, 새로운 데이터에 일반화하지 못함
> 
> 해결: Out-of-Fold Target Encoding — 자기 자신의 정답을 절대 참조하지 못하게 차단

### 2.5 가설 검정 — "이것이 우연인가, 진짜 패턴인가?"

> **이전 연결**: 2.4에서 데이터로 모집단을 추측하는 법을 배웠다.
> **이 절의 목적**: "이 추측이 **통계적으로 신뢰할 수 있는가**"를 판단하는 프레임워크를 배운다.
> **음악 프로젝트 연결**: "스킵률 증가 → 재생량 감소" 가설을 검증한 바로 그 도구다.

#### 📌 용어 해설
- **귀무가설(null hypothesis, H₀)**: "아무 효과 없다"는 기본 가정. 디폴트 = "우연이다"
- **대립가설(alternative hypothesis, H₁)**: "효과가 있다"는 주장. 우리가 증명하고 싶은 것
- **p-value (피 밸류)**: "귀무가설이 맞다고 가정할 때, 현재 데이터처럼 극단적인 결과가 우연히 나올 확률"
  - p-value가 작으면 → "이건 우연으로 보기 힘들다" → 귀무가설 기각
- **유의수준 α (알파)**: "이 정도면 충분히 작다"고 판단하는 기준선 (보통 0.05 = 5%)

**절차 (with 음악 프로젝트 예시):**

```
1단계) 가설 설정
  H₀: "스킵률 변화와 재생량 변화 사이에 상관관계가 없다" (ρ = 0)
  H₁: "음의 상관관계가 있다" (ρ < 0, 스킵이 늘면 재생량이 줄어든다)

2단계) 데이터 수집
  6명의 유저, 수천 곡의 전반부/후반부 스킵률 & 재생량 변화 데이터

3단계) 검정통계량 계산
  Spearman ρ = -0.23, p-value = 0.00001

4단계) 판단
  p-value (0.00001) < α (0.05)
  → "이 정도의 음의 상관관계가 우연히 나올 확률은 0.001%"
  → 우연으로 보기 힘들다 → H₀ 기각 → "진짜 패턴이다" ✅
```

**실제 코드 (regression_analysis.py):**

```python
# Spearman 순위 상관계수와 p-value 계산
spearman_rho, p_value_s = stats.spearmanr(df_reg['skip_delta'], df_reg['log_change'])

if p_value_s < 0.05:
    print("✅ 통계적으로 유의미한 결과입니다.")
    print("   '스킵률 증가'는 미래 '재생량 감소'와 확실한 음의 상관관계를 가짐을 증명합니다.")
```

**Pearson vs Spearman — 둘 다 "상관계수"인데, 무엇이 다른가?**

두 변수 사이에 "관계가 있는가?"를 측정하는 지표다. 결과는 -1 ~ +1 사이 값.

```
+1: 완벽한 양의 관계 (X 증가 → Y도 증가)
 0: 관계 없음
-1: 완벽한 음의 관계 (X 증가 → Y 감소)
```

#### Pearson r — "직선 위에 얼마나 가까이 있는가"

```
수식:
  r = Σ[(xᵢ - x̄)(yᵢ - ȳ)] / √[Σ(xᵢ - x̄)² × Σ(yᵢ - ȳ)²]

쉽게 말해: "X와 Y를 그래프에 찍었을 때, 점들이 직선에 가까우면 |r| → 1"
```

#### Spearman ρ — "순위가 일치하는가"

```
방법:
  1. X 값들을 크기 순서로 순위를 매긴다 (1등, 2등, 3등...)
  2. Y 값들도 크기 순서로 순위를 매긴다
  3. "순위"에 대해 Pearson r을 계산한다

핵심 차이: 원래 값이 아니라 "순위"를 사용 → 이상치에 강건
```

#### 🔢 수학 예시 — 5곡의 스킵률과 재생량 변화

```
데이터:
  곡     스킵률변화(X)   재생량변화(Y)
  곡1     -0.2           +0.5        (스킵 줄었고, 재생량 늘었음)
  곡2     +0.1           -0.1        (스킵 약간 늘었고, 재생량 약간 줄었음)
  곡3     +0.5           -0.8        (스킵 많이 늘었고, 재생량 크게 줄었음)
  곡4     +0.05          +0.02       (변화 거의 없음)
  곡5     +0.3           -0.3        (스킵 늘었고, 재생량 줄었음)
```

**Pearson r 계산:**

```
  원래 값 그대로 사용:
  x̄ = (-0.2+0.1+0.5+0.05+0.3)/5 = 0.15
  ȳ = (0.5-0.1-0.8+0.02-0.3)/5 = -0.136
  
  (생략하고 결과만) → r = -0.97  
  → -1에 매우 가까움 = 강한 음의 직선 관계
```

**Spearman ρ 계산:**

```
  Step 1: 순위로 변환
  곡     X 원래값  X 순위    Y 원래값  Y 순위
  곡1    -0.2      1등(최소)  +0.5     5등(최대)
  곡2    +0.1      3등       -0.1     3등
  곡3    +0.5      5등(최대)  -0.8     1등(최소)
  곡4    +0.05     2등       +0.02    4등
  곡5    +0.3      4등       -0.3     2등

  Step 2: 순위에 대해 Pearson 계산
  X순위: [1, 3, 5, 2, 4]
  Y순위: [5, 3, 1, 4, 2]
  
  → ρ = -1.0  (순위가 완벽히 반대!)
  → X 순위 1등인 곡이 Y 순위 5등, X 순위 5등인 곡이 Y 순위 1등
  → "스킵이 가장 적은 곡이 재생량 증가가 가장 크다" — 완벽한 역순위 관계
```

**비교 정리:**

| | Pearson r | Spearman ρ |
|---|---|---|
| **계산 대상** | 원래 값 | 순위로 변환한 값 |
| **측정하는 것** | "직선 관계"의 강도 | "순위 일치도" (단조 관계) |
| **이상치에** | 민감 (값 하나가 크면 결과 왜곡) | 강건 (순위는 이상치 영향 적음) |
| **언제 쓰나** | 데이터가 깔끔할 때 | 분포가 찌그러졌을 때 ✅ |

```
이상치가 있으면 어떻게 되나:

  만약 곡3의 재생량변화가 -0.8이 아니라 -50.0 (극단적 이상치)이라면?
  
  Pearson: r이 크게 달라짐 (이상치 하나에 끌려감) ❌
  Spearman: 곡3의 Y순위는 여전히 "1등(최소)" → ρ 변화 없음 ✅
```

> [!TIP]
> regression_analysis.py에서 Pearson **과** Spearman **둘 다** 계산한 이유:
> Spearman이 더 강건하지만, Pearson도 같이 보면 "선형적으로도 관계가 있는지" 추가 확인 가능.
> 두 지표가 모두 유의하면 → 더 강력한 증거.

### 2.6 다중 비교 문제

> **왜 이것을 알아야 하는가?**
> KKBox에서 100개 이상의 피처를 만들어 테스트하면, α=0.05라도 **평균 5개**는 우연히 유의하게 나올 수 있다. "많이 시도하면 우연한 성공도 생긴다"는 함정.

```
비유: 동전을 20번 던져서 "이 동전은 편향되어 있다!"고 주장하는 것은 어렵다.
     하지만 100개의 동전을 각각 20번씩 던지면?
     → 순전히 운으로, 5개쯤은 "앞면이 75% 이상" 나올 수 있다.
     → 이 5개만 보고 "편향된 동전 발견!"이라고 하면 거짓양성(false positive)

보정 방법:
  Bonferroni: α를 테스트 횟수로 나눔 (100번 테스트면 α = 0.05/100 = 0.0005)
              → 매우 보수적 (진짜 효과도 놓칠 수 있음)
  
  Benjamini-Hochberg (FDR): "거짓양성 비율"을 제어 → 더 현실적
```

## Chapter 3. 선형회귀 — "직선으로 예측하기"

> **"선형회귀를 제대로 이해하면, 모든 통계적 학습의 기초가 잡힌다."**
> — ESL Ch.3, CS229 Lecture 2

### 🧭 왜 이것을 배우는가?

음악 프로젝트에서 이런 질문을 했다:
> "전반부에 스킵률이 10% 증가하면, 후반부 재생량은 평균적으로 얼마나 감소하는가?"

이 질문에 답하는 가장 기본적인 방법이 **선형회귀**다. `regression_analysis.py`에서 `stats.linregress()`를 호출한 것이 바로 이것.

선형회귀는 또한 **모든 복잡한 모델의 조상**이다:
- 로지스틱 회귀 = 선형회귀 + 시그모이드 함수
- 신경망의 각 뉴런 = 선형회귀 + 활성함수
- 트리의 각 리프 = "국소적 평균" (0차 선형회귀)

---

### 3.1 단순선형회귀 — "X 하나로 Y 예측"

```
y = β₀ + β₁x + ε

y:  예측하고 싶은 값 (반응변수, 종속변수)
x:  예측에 사용하는 값 (설명변수, 독립변수)
β₀: 절편 (x=0일 때의 y값)
β₁: 기울기 (x가 1 증가할 때 y의 변화량)
ε:  오차 (모델이 설명 못하는 나머지, 노이즈)
```

**구체적 예시 — regression_analysis.py에서의 실제 분석:**

```python
slope, intercept, r_value, p_value, std_err = stats.linregress(
    df_reg['skip_delta'], df_reg['log_change']
)
# r_value = Pearson 상관계수 (Ch.2.5에서 배운 것)
# R² = r_value² = 설명력
```

```
X = skip_delta (스킵률 변화량)
  -0.5 = 전반부보다 후반부에 스킵이 50%p 줄었다 (점점 좋아하는 중)
   0   = 스킵률 변화 없음
  +0.3 = 전반부보다 후반부에 스킵이 30%p 늘었다 (질려가는 중)

Y = log_change (재생량 변화, log2 스케일)
  -1 = 후반부 재생량이 전반부의 절반 (반토막)
   0 = 변화 없음
  +1 = 후반부 재생량이 전반부의 2배

결과: Y = -0.85 × X + 0.02

해석:
  "스킵률이 10%p 증가하면(더 많이 넘기면), 
   재생량은 평균적으로 2^(-0.085) - 1 ≈ 5.7% 감소한다"

  기울기(slope) = -0.85 → 음수 = 스킵 증가 시 재생량 감소 (가설 확인!)
  절편(intercept) = 0.02 → 스킵 변화 없을 때 재생량 거의 유지
  r_value = -0.28 → Pearson 상관계수 (약한 음의 상관)
```

**R²(결정계수)는 어떻게 나왔나:**

```
R² = r_value² = (-0.28)² = 0.08

R²의 의미: "X(스킵률)가 Y(재생량 변화)의 변동을 얼마나 설명하는가?"

구체적으로:
  Y의 전체 변동 = 100%라고 하자 (모든 곡의 재생량 변화가 제각각)
  
  이 중:
    스킵률 변화로 설명되는 부분 = 8%   ← R²
    설명 안 되는 나머지         = 92%  ← 기분, 날씨, 상황, 다른 요인들

수식으로:
  R² = 1 - (잔차 변동) / (전체 변동)
     = 1 - Σ(yᵢ - ŷᵢ)² / Σ(yᵢ - ȳ)²

  ŷᵢ = 모델 예측값 (직선 위의 점)
  ȳ  = Y의 평균
  
  분자: "직선에서 벗어난 정도" (작을수록 좋음)
  분모: "애초에 Y가 얼마나 흩어져 있었는지"
  
  R² = 0:   직선이 아무것도 설명 못함 (그냥 평균 쓰는 거랑 같음)
  R² = 0.5: 전체 변동의 50%를 직선이 설명
  R² = 1:   모든 점이 직선 위 (완벽한 설명)
```

> [!NOTE]
> R² = 0.08이 "낮아 보이지만", 음악 청취 행동은 기분/날씨/상황 등 수많은 요인의 영향을 받으므로,
> **스킵률 하나만으로** 8%를 설명하는 것은 의미 있는 결과다.
> "약하지만 확실한 관계"가 있다는 뜻. (p-value가 0.05 미만이므로 우연이 아님)

### 3.2 다중선형회귀 — "여러 X로 Y 예측"

```
y = β₀ + β₁x₁ + β₂x₂ + ... + βₙxₙ + ε

행렬 표현:
  y = Xβ + ε

  y: n개의 관측값 벡터
  X: n×(d+1) 행렬 (첫 열은 전부 1 — 절편용)
  β: (d+1)개의 계수 벡터
  ε: n개의 오차 벡터
```

```
β̂ = (XᵀX)⁻¹Xᵀy   ← 이것이 "정규방정식(Normal Equation)"

의미: 데이터에 가장 잘 맞는 직선(초평면)의 기울기와 절편을 
     한 번의 행렬 계산으로 구할 수 있다! (반복 계산 필요 없음)
```

#### 🔢 수학 예시 — 최소제곱 해를 두 가지 방법으로 구하기

3개의 데이터 포인트로 Y = β₀ + β₁X 를 구해보자:

```
  X (스킵률변화)    Y (재생량변화)
       1                1
       2                2
       3                2
```

**방법 1: 편미분으로 풀기 (미적분 접근)**

```
Step 1: 손실함수 = 오차의 제곱합
  L(β₀, β₁) = (1 - β₀ - β₁·1)² + (2 - β₀ - β₁·2)² + (2 - β₀ - β₁·3)²

Step 2: β₀에 대해 편미분 = 0 (Ch.1.5의 편미분!)
  ∂L/∂β₀ = -2[(1-β₀-β₁) + (2-β₀-2β₁) + (2-β₀-3β₁)] = 0
  
  정리: 5 - 3β₀ - 6β₁ = 0
  →  3β₀ + 6β₁ = 5   ... (식 ①)

Step 3: β₁에 대해 편미분 = 0
  ∂L/∂β₁ = -2[1·(1-β₀-β₁) + 2·(2-β₀-2β₁) + 3·(2-β₀-3β₁)] = 0
  
  정리: 11 - 6β₀ - 14β₁ = 0
  →  6β₀ + 14β₁ = 11  ... (식 ②)

Step 4: 연립방정식 풀기
  ①×2:  6β₀ + 12β₁ = 10
  ②  :  6β₀ + 14β₁ = 11
  ②-①×2:     2β₁ = 1  →  β₁ = 0.5
  ①에 대입: 3β₀ + 3 = 5  →  β₀ = 2/3 ≈ 0.667

결과: Y = 0.667 + 0.5X ✅
```

**방법 2: 행렬로 풀기 (정규방정식)**

```
Step 1: 설계행렬 X와 Y 벡터 만들기 (첫 열에 1 추가 — 절편용)

  X = [ 1  1 ]      y = [ 1 ]
      [ 1  2 ]          [ 2 ]
      [ 1  3 ]          [ 2 ]

Step 2: XᵀX 계산 (2×3 · 3×2 = 2×2)

  Xᵀ = [ 1  1  1 ]
       [ 1  2  3 ]

  XᵀX = [ 1+1+1    1+2+3 ]   [ 3   6 ]
        [ 1+2+3    1+4+9 ] = [ 6  14 ]

Step 3: Xᵀy 계산 (2×3 · 3×1 = 2×1)

  Xᵀy = [ 1·1+1·2+1·2 ]   [  5 ]
        [ 1·1+2·2+3·2 ] = [ 11 ]

Step 4: (XᵀX)⁻¹ 역행렬 구하기

  2×2 역행렬 공식: [a b]⁻¹ = 1/(ad-bc) × [ d -b]
                   [c d]                   [-c  a]
  
  det = 3×14 - 6×6 = 42 - 36 = 6

  (XᵀX)⁻¹ = (1/6) × [ 14  -6 ] = [ 7/3   -1  ]
                      [ -6   3 ]   [  -1   1/2 ]

Step 5: β̂ = (XᵀX)⁻¹ × Xᵀy

  β̂ = [ 7/3  -1  ] × [  5 ]
      [  -1  1/2 ]   [ 11 ]

  β₀ = 7/3 × 5 + (-1) × 11 = 35/3 - 33/3 = 2/3 ≈ 0.667
  β₁ = (-1) × 5 + 1/2 × 11 = -5 + 5.5 = 0.5

결과: Y = 0.667 + 0.5X ✅ (편미분과 동일!)
```

**두 방법이 같은 이유:**

```
편미분에서 나온 연립방정식:  3β₀ +  6β₁ =  5
                            6β₀ + 14β₁ = 11

이것을 행렬로 쓰면:  [ 3   6 ] [β₀]   [ 5 ]
                     [ 6  14 ] [β₁] = [11 ]

                     =  (XᵀX)  × β  =  Xᵀy   ← 정규방정식 그 자체!

→ 행렬 방법은 편미분의 "행렬 버전"일 뿐, 수학적으로 완전히 동일하다.
   차이는 편의성: 피처가 100개여도 β̂ = (XᵀX)⁻¹Xᵀy 한 줄로 표현 가능.
```

**구체적 예시 — "만약 KKBox에서 선형회귀를 쓴다면?":**

```
  재청취 확률 = 0.3 × (유저 활동량) 
              + 0.2 × (곡 인기도) 
              + 0.5 × (유저-아티스트 친밀도) 
              - 0.1 × (곡 길이) 
              + 0.15

  → 유저 활동량이 높고, 인기 있는 곡이고, 좋아하는 아티스트이며, 
    곡이 짧을수록 다시 들을 확률이 높다
```

> [!NOTE]
> **왜 KKBox에서 선형회귀를 직접 쓰지 않는가?**
> 1. 재청취 확률은 0~1 사이여야 하는데, 선형회귀는 -∞~+∞까지 값이 나올 수 있다
> 2. 피처 간 **비선형 상호작용**이 복잡하다 (아티스트 친밀도 × 곡 길이 × 시간대)
> 3. **비선형 패턴**이 있다 (아티스트 재생 50회까지는 ↑, 그 이후는 ↓ — "질림")
> 
> 하지만 선형회귀의 원리는 **모든 모델의 기초**다:
> - LightGBM의 각 leaf 값 = 국소적 평균 (가장 단순한 선형회귀)
> - 신경망의 각 뉴런 = `w₁x₁ + w₂x₂ + ... + b` (바로 선형 결합!)

---

### 3.3 선형회귀의 가정 — "이 조건이 깨지면 결과를 못 믿는다"

OLS가 **BLUE (Best Linear Unbiased Estimator, 최적 선형 불편 추정량)** 가 되려면:

| 가정 | 의미 | KKBox에서 깨지는 예시 |
|------|------|---------------------|
| **선형성** | y와 X의 관계가 직선 | 아티스트 재생 횟수↑ → 재청취 확률이 ↑했다가 ↓ (비선형) |
| **등분산성** | 오차의 크기가 일정 | 인기곡은 변동 작고, 비인기곡은 변동 큼 |
| **무자기상관** | 오차끼리 관계 없음 | 같은 유저의 연속 곡들은 서로 영향 (시퀀스 효과) |
| **공선성 없음** | X들끼리 독립 | user_count와 user_unique_songs는 매우 높은 상관 |

> 가정이 깨지면? → 선형회귀의 추정값을 신뢰할 수 없다.
> **해결책**: ① 비선형 모델 사용 (트리, 신경망) ② 데이터 변환 ③ 정규화 추가

### 3.4 결정계수 R² — "모델이 데이터를 얼마나 설명하는가"

```
R² = 1 - (잔차 제곱합) / (전체 분산)
   = 1 - Σ(yᵢ - ŷᵢ)² / Σ(yᵢ - ȳ)²

R² = 0:   모델이 전혀 설명 못함 (그냥 평균 쓰는 것과 동일)
R² = 0.5: 전체 변동의 50%를 설명
R² = 1:   완벽한 설명 (모든 점이 직선 위)
```

**조정 R² — "피처를 늘렸다고 좋아진 거 맞아?"**

```
조정 R² = 1 - [(1-R²)(n-1)] / [n-d-1]

n: 데이터 수, d: 피처 수

문제: 쓸모없는 피처를 추가해도 R²는 절대 감소하지 않는다
해결: 조정 R²는 피처 수에 패널티를 부과 → 의미 없는 피처 추가 시 감소
```

## Chapter 4. 로지스틱 회귀와 분류

> 📕 참조: ESL Ch.4, CS229 Lecture 3, PRML Ch.4

### 🧭 왜 이것을 배우는가?

KKBox 대회의 목표는 **"재청취할까 안 할까"**를 예측하는 것이다.
- 선형회귀는 "몇 점?" (연속 값)에 답한다
- 로지스틱 회귀는 **"이거 맞아 안 맞아?" (이진 분류)**에 답한다

이것은 Chapter 3에서 배운 선형회귀에 **하나의 변환**만 추가한 것이다.

---

### 4.1 로지스틱 회귀 — "확률로 변환하기"

#### 📌 용어 해설
- **시그모이드 함수(sigmoid function)**: 어떤 숫자든 0~1 사이로 눌러 넣는 S자형 함수. "확률 변환기"
- **로그 오즈(log-odds)**: "성공 확률 / 실패 확률"에 로그를 씌운 것

```
선형회귀의 문제: y = wx + b → 결과가 -∞ ~ +∞ (확률이 아님!)

해결: 시그모이드 함수를 씌운다
  σ(z) = 1 / (1 + e⁻ᶻ)    여기서 z = wx + b

  z = -5 → σ(-5) = 0.007 ≈ 0%   (거의 확실히 아님)
  z =  0 → σ(0)  = 0.500 = 50%  (반반)
  z = +5 → σ(5)  = 0.993 ≈ 100% (거의 확실히 맞음)
```

**그래프 모양:**

```
확률
1.0 |                          ──────────
    |                        /
    |                      /
0.5 |..................../.................
    |                  /
    |                /
0.0 |──────────────
    └────────────────────────────────── z (= wx + b)
         -5       0        +5
```

**구체적 예시 — "KKBox에서의 로지스틱 회귀":**

```
입력 피처:
  user_artist_ratio = 0.12 (유저가 이 아티스트를 12% 비율로 들음)
  song_count = 342 (이 곡을 총 342명이 들었음)
  svd_dot = 2.34 (유저-곡 궁합 높음)

선형 결합:
  z = 0.5 × 0.12 + 0.01 × 342 + 0.3 × 2.34 + (-0.5)
  z = 0.06 + 3.42 + 0.70 - 0.5
  z = 3.68

시그모이드 변환:
  P(재청취) = 1 / (1 + e⁻³·⁶⁸) = 0.975 = 97.5%

→ "이 유저는 이 곡을 다시 들을 확률이 97.5%"
```

---

### 4.2 손실함수: Binary Cross-Entropy — "모델이 얼마나 틀렸는가"

```
L = -(1/n) × Σ [yᵢ × log(pᵢ) + (1-yᵢ) × log(1-pᵢ)]

yᵢ: 실제 정답 (0 또는 1)
pᵢ: 모델이 예측한 확률 (0~1)
L : 손실값 (낮을수록 좋다)
```

**이것이 KKBox LightGBM의 `'objective': 'binary'`가 최적화하는 함수다.**

**구체적 예시 — "손실이 어떻게 작동하는지":**

```
케이스 1: 정답=재청취(y=1), 모델 예측=0.95
  L₁ = -[1 × log(0.95) + 0 × log(0.05)]
     = -log(0.95) = 0.05  ← 작은 손실 (잘 맞췄으니까!) ✅

케이스 2: 정답=재청취(y=1), 모델 예측=0.10
  L₂ = -[1 × log(0.10) + 0 × log(0.90)]
     = -log(0.10) = 2.30  ← 큰 손실 (놓쳤으니까!) ❌

케이스 3: 정답=안들음(y=0), 모델 예측=0.95
  L₃ = -[0 × log(0.95) + 1 × log(0.05)]
     = -log(0.05) = 3.00  ← 매우 큰 손실 (완전히 틀렸으니까!) ❌❌

직관: "자신있게 틀리면 엄청난 페널티를 받는다"
```

---

### 4.3 평가 지표: AUC-ROC

> [!IMPORTANT]
> KKBox 대회의 평가 지표는 **AUC (Area Under the ROC Curve)**이다.

#### 📌 용어 해설
- **ROC 곡선**: "임계값(threshold)을 바꿔가며 그린 True Positive Rate vs False Positive Rate 그래프"
- **AUC**: ROC 곡선 아래의 면적. 0.5(랜덤)~1.0(완벽) 사이 값

**AUC를 가장 쉽게 이해하는 방법:**

```
AUC = "모델이 양성(다시 들은 곡)과 음성(안 들은 곡)을 
       얼마나 잘 구분하는가?"

실험: 재청취한 곡(양성) 하나와 재청취 안 한 곡(음성) 하나를 
     무작위로 뽑는다.

질문: 모델이 양성 곡에 더 높은 점수를 줄 확률은?

AUC = 0.50 → 반반. 동전 던지기. 모델이 아무것도 못 배운 것.
AUC = 0.65 → 65%. KKBox 초기 파이프라인 수준.
AUC = 0.67 → 67%. KKBox v8 Kaggle 제출 수준.
AUC = 0.85 → 85%. KKBox v6a OOF 수준 (로컬, 좀 낙관적).
AUC = 1.00 → 100% 완벽. 세상에 존재하지 않는 수준.
```

**구체적 예시 — KKBox AUC 변화 추이:**

```
v3 (기본 메타데이터만):      Kaggle AUC = 0.58  "겨우 랜덤보다 나은 수준"
v4 (Target Encoding 누수):  로컬 0.96, Kaggle 0.58 "자기기만의 극치"
v6a (OOF 수정 + Count 피처): Kaggle AUC = 0.67  "드디어 의미 있는 성능"
v8 (SVD + 시퀀스):           Kaggle AUC ≈ 0.67+ "피처 품질로 조금 더"
```

### 4.4 정밀도, 재현율, F1 — "추천 시스템의 관점"

```
         예측: 다시 듣는다     예측: 안 듣는다
실제 다시 들음    TP (맞음!)        FN (놓침)
실제 안 들음      FP (오추천)       TN (맞음!)

정밀도(Precision) = TP / (TP + FP)
  "추천한 곡 10곡 중 실제로 다시 들은 곡은 몇 곡?" 
  → 7/10 = 70% → "상위 추천의 질(quality)"

재현율(Recall) = TP / (TP + FN)
  "유저가 실제로 다시 들은 모든 곡 중, 내 추천에 포함된 비율"
  → 7/20 = 35% → "추천의 커버리지(coverage)"

F1 = 2 × (정밀도 × 재현율) / (정밀도 + 재현율)
  → 둘의 균형
```

> [!TIP]
> 음악 추천 실무에서는 **Precision@K**가 더 유용하다.
> "상위 10곡(K=10)을 추천했을 때, 실제로 유저가 다시 들은 곡이 몇 개인가?"
> → 유저는 추천 목록의 위에서부터 보니까, 상위 몇 개가 얼마나 정확한지가 핵심.

## Chapter 5. 정규화 — "모델이 답을 외우지 못하게 막는 법"

> **"좋은 모델은 데이터에 잘 맞으면서도 단순해야 한다."**
> — 오컴의 면도날 (Occam's Razor)
> 📕 참조: ESL Ch.3.4, CS229 Lecture 7

### 🧭 왜 이것을 배우는가?

KKBox v4에서의 참사를 기억하자:
- 로컬 AUC: **0.96** (거의 완벽!)
- Kaggle AUC: **0.58** (랜덤보다 약간 나은 수준)

이것이 **과적합(overfitting)**이다 — 모델이 학습 데이터의 "패턴"이 아니라 "정답 자체"를 외워버린 것.
정규화(Regularization)는 이런 과적합을 **방지하는 모든 기법**을 통칭한다.

---

### 5.1 과적합의 본질: 편향-분산 트레이드오프

#### 📌 용어 해설
- **과적합(overfitting)**: 학습 데이터에서는 정확하지만, 새로운 데이터에서는 부정확한 상태. "시험 족보만 외운 학생"
- **과소적합(underfitting)**: 학습 데이터에서조차 부정확한 상태. "공부를 안 한 학생"
- **편향(bias)**: 모델이 체계적으로 틀리는 정도. "항상 같은 방향으로 빗나감"
- **분산(variance)**: 모델이 데이터에 따라 변하는 정도. "할 때마다 결과가 달라짐"

```
총 오차 = 편향² + 분산 + 노이즈(줄일 수 없는 근본적 불확실성)

비유 — 양궁 선수:
  높은 편향 + 낮은 분산 = 화살이 항상 왼쪽 위에 모여있음 (체계적 오류)
  낮은 편향 + 높은 분산 = 화살이 과녁 중심 근처에 넓게 흩어져 있음 (불안정)
  낮은 편향 + 낮은 분산 = 화살이 중심에 모여있음 (이상적) ✅
```

**KKBox 사례:**

| 상태 | 증상 | KKBox 사례 |
|------|------|-----------|
| **과소적합** (편향 高) | Train도 나쁘고 Val도 나쁨 | 선형 로지스틱 회귀만 사용 |
| **과적합** (분산 高) | Train은 좋은데 Val은 나쁨 | v4 Target Encoding 누수 (0.96 vs 0.58) |
| **적정** | Train과 Val 모두 적당히 좋음 | v6a 이후 OOF 검증 (0.85 vs 0.67) |

---

### 5.2 L2 정규화 (Ridge) — "계수를 작게 유지해라"

#### 📌 용어 해설
- **계수(coefficient)**: 여기서 말하는 계수는 **회귀 모델의 β값** (기울기)이다. **상관계수(r)와 다른 것!**
  - 계수(β): Y = **-0.85** × X + 0.02 에서 -0.85. "X가 1 변하면 Y가 몇 변하는가"
  - 상관계수(r): -1~+1 사이. "X와 Y가 함께 움직이는 정도" (Ch.2.5)
- **Ridge 회귀**: 손실함수에 "계수(β)의 제곱합"을 패널티로 추가. 계수가 클수록 페널티 증가
- **L2 노름(norm)**: 벡터의 각 원소를 제곱해서 합한 뒤 루트. 벡터의 "길이" 개념

#### 왜 "큰 계수"가 나쁜가 — 구체적 예시

```
데이터 5개로 "스킵률 → 재생량" 모델을 만들었다:

모델 A (계수가 작은 모델):
  재생량 = -0.8 × 스킵률 + 0.5
  β₁ = -0.8
  → 스킵률 10% 변하면 재생량 0.08 변화 (안정적)

모델 B (계수가 큰 모델):
  재생량 = -254.3 × 스킵률 + 187.2 × 스킵률² - 42.8 × 스킵률³ + 0.01
  β₁ = -254.3, β₂ = 187.2, β₃ = -42.8
  → 학습 데이터 5개를 완벽히 통과 (오차 = 0!)
  → 하지만 스킵률이 0.01만 달라져도 예측이 크게 출렁임

  * 참고: X², X³ 같은 항을 추가하면 곡선 표현이 가능해진다.
    이것은 "다항 회귀"인데, X₁=스킵률, X₂=스킵률², X₃=스킵률³으로
    놓으면 여전히 Y = β₁X₁ + β₂X₂ + β₃X₃ (다중선형회귀) 형태다.

왜 모델 B가 위험한가:
  스킵률=0.10 → 예측: 0.32 (학습 데이터에 있던 값 → 정확)
  스킵률=0.11 → 예측: 2.87 (살짝 다른데 결과가 급변!)
  스킵률=0.12 → 예측: -1.54 (음수?! 새 데이터에서 엉망)

  → 계수가 크면, 입력의 아주 작은 변화에도 출력이 극단적으로 출렁인다
  → 학습 데이터의 잡음까지 외운 것 = 과적합
```

#### Ridge가 하는 일

```
원래 목표:  "오차를 최소화해라"
  min Σ(yᵢ - ŷᵢ)²
  → 오차만 줄이려고 하면, 계수를 극단적으로 키워서라도 데이터에 끼워맞춤

Ridge 목표: "오차를 최소화하되, 계수가 크면 벌금을 내라"
  min Σ(yᵢ - ŷᵢ)² + λ × (β₁² + β₂² + ... + βₙ²)
      ──────────     ────────────────────────────
       오차 줄이기         벌금 (계수가 클수록 커짐)

λ (람다): 벌금의 강도
  λ = 0   → 벌금 없음 (원래 선형회귀)       → 과적합 위험
  λ = 적당 → 진짜 중요한 계수는 유지, 잡음은 억제 → 적절 ✅  
  λ → ∞   → 벌금이 너무 커서 모든 계수가 0   → 과소적합
```

```
숫자로 보면:

  모델 B의 계수: β = [-254.3, 187.2, -42.8]
  계수의 제곱합 = 254.3² + 187.2² + 42.8² = 64,648 + 35,044 + 1,832 = 101,524
  → λ=0.01만 해도 벌금 = 0.01 × 101,524 = 1,015 (엄청 큰 벌금!)

  모델 A의 계수: β = [-0.8]
  계수의 제곱합 = 0.8² = 0.64
  → 벌금 = 0.01 × 0.64 = 0.006 (벌금 거의 없음)

  → Ridge는 자연스럽게 모델 A(단순)를 선호하고, 모델 B(복잡)를 벌한다!
```

**"실제로 계수가 커야 하면 어떡해?"**

```
만약 진짜로 β₁ = 50이어야 하는 상황이라면:
  Ridge는 β₁을 약간 줄인다 (예: 50 → 45) → 약간의 편향(bias) 발생
  하지만 잡음에 의한 불안정성(분산)이 크게 줄어든다

  총 오차 = 편향² + 분산  (5.1에서 배운 것)
  → 편향이 조금 늘더라도, 분산이 크게 줄면 총 오차는 감소!
  → λ를 교차검증(5.4)으로 적절히 정하면, 이 균형을 최적화할 수 있다
```

**KKBox LightGBM에서:**

```python
params = {
    'reg_lambda': 5.0,   # ← 이것이 L2 정규화 (Ridge)
}
```

→ 각 리프의 출력값(예측값)이 과도하게 극단적이면 페널티를 부과.

#### Ridge의 핵심 직관 정리

```
OLS와 Ridge는 "다른 문제"를 푸는 것이다:

  OLS:    β̂ = (XᵀX)⁻¹Xᵀy          ← "오차만 최소화"의 유일한 답
  Ridge:  β̂ = (XᵀX + λI)⁻¹Xᵀy     ← "오차 + 벌금 최소화"의 유일한 답
                      ↑
                  이것 때문에 다른 답이 나옴
```

```
왜 Ridge가 필요한가 — 노이즈 민감도:

  β = 200일 때:
    X가 0.01만 달라져도 → Y가 200 × 0.01 = 2.0 변함
    → 노이즈가 0.01만 섞여도 결과가 2.0이나 출렁임 ❌

  β = 0.8일 때:
    X가 0.01 달라져도 → Y가 0.8 × 0.01 = 0.008 변함
    → 노이즈에 거의 안 흔들림 ✅

  Ridge의 선택:
    "학습 데이터에서 조금 더 틀리더라도 (편향 ↑),
     새 데이터의 노이즈에 흔들리지 않도록 (분산 ↓) 
     계수를 작게 유지하겠다."
```

### 5.3 L1 정규화 (Lasso) — "쓸모없는 피처를 자동으로 제거"

```
Lasso 손실 = (원래 손실) + λ × (모든 계수의 절대값 합)
           = Σ(yᵢ - ŷᵢ)² + λ × (|β₁| + |β₂| + ... + |βₙ|)
```

**Ridge vs Lasso — 핵심 차이:**

```
Ridge (L2): 모든 계수를 "0에 가깝게" 줄이지만, 정확히 0으로 만들진 않음
  → 100개 피처가 있으면 100개 전부 약간씩 사용

Lasso (L1): 불필요한 계수를 "정확히 0"으로 만들어 버림
  → 100개 피처 중 쓸모없는 70개의 계수 = 0 → 자동 피처 선택!

기하학적 직관:
  Ridge = 원형 제약 영역 → 해가 원의 둘레 아무 곳에서나 나올 수 있음
  Lasso = 마름모형 제약 영역 → 해가 꼭짓점(축 위)에서 나올 확률이 높음
        → 꼭짓점 = 특정 계수가 정확히 0인 지점!
```

**KKBox LightGBM에서:**

```python
params = {
    'reg_alpha': 1.0,    # ← 이것이 L1 정규화 (Lasso)
}
```

→ 쓸모없는 분할을 억제. KKBox v8에서 `reg_alpha=1.0` + `reg_lambda=5.0`은 사실상 **Elastic Net** (Ridge + Lasso 결합).

#### 왜 Lasso는 계수를 "정확히 0"으로 만들 수 있는가?

각 계수에는 두 힘이 작용한다: **데이터가 살리려는 힘** vs **벌금이 죽이려는 힘**.

```
예: 데이터가 β = 0.3을 원하는 약한 피처, λ = 1.0

  β가 어디에 있든:
    데이터의 힘 = 2×(0.3 - β)    ← β를 0.3으로 당기는 힘 (β에 따라 변함)
    벌금의 힘   = λ = 1.0        ← β를 0으로 당기는 힘 (항상 일정!)

  β = 0에서 두 힘을 비교:
    데이터: 2×(0.3 - 0) = 0.6   ← "0.3으로 올려!"
    벌금:              = 1.0    ← "0으로 줄여!"
    
    0.6 < 1.0 → 벌금이 이김! → β = 0에서 못 벗어남 🔒

  → 데이터의 최대 힘(0.6)이 벌금(1.0)보다 작으니까,
    이 피처의 계수는 영원히 0으로 고정된다 (제거됨!)
```

```
반대로: 데이터가 β = 5.0을 원하는 강한 피처, λ = 1.0

  β = 0에서:
    데이터: 2×(5.0 - 0) = 10.0  ← "5.0으로 올려!!"
    벌금:              = 1.0   ← "0으로 줄여!"
    
    10.0 > 1.0 → 데이터가 이김! → β는 살아남음 ✅
    (다만 벌금 때문에 5.0이 아니라 4.5 정도에서 멈춤)
```

```
Ridge에서는 왜 안 되는가:
  Ridge 벌금의 힘 = 2λβ → β가 작아지면 벌금 힘도 같이 약해짐
  
  β = 0.1 → 벌금 힘 = 0.2   (약함)
  β = 0.01 → 벌금 힘 = 0.02 (더 약함)
  → 어딘가에서 "데이터 힘 = 벌금 힘" 균형 → β = 0.04 같은 작은 값에서 멈춤
  → 0에 가까워지지만 정확히 0이 되진 않음
```

**λ에 따른 제거 범위:**

```
  λ 작음 → 아주 약한 피처만 제거 (대부분 생존)
  λ 중간 → 중간 피처도 제거 (절반 사라짐)
  λ 큼   → 가장 강한 피처만 생존 (대부분 사라짐)
  λ → ∞  → 전부 사라짐 (아무것도 예측 못함)
```

### 5.4 교차 검증 — "시험지를 바꿔가며 실력 확인"

> **이 절이 왜 정규화 챕터에 있는가?**
> 정규화의 강도(λ)를 "얼마나 세게 할지" 결정하려면, 모델의 실제 성능을 정확히 측정해야 한다.
> 교차 검증이 바로 그 측정 도구다.

#### 📌 용어 해설
- **K-Fold 교차 검증**: 데이터를 K조각으로 나눠서, 각 조각을 한 번씩 "시험지"로 사용
- **OOF (Out-of-Fold)**: "내가 학습할 때 보지 않은 데이터로 성능을 측정"
- **Stratified**: 각 폴드에서 target=1/0의 비율을 동일하게 유지

```
5-Fold 교차 검증:

  전체 데이터를 5등분:  [A] [B] [C] [D] [E]

  Fold 1: 학습=[B,C,D,E] → 검증=[A] → AUC = 0.84
  Fold 2: 학습=[A,C,D,E] → 검증=[B] → AUC = 0.86
  Fold 3: 학습=[A,B,D,E] → 검증=[C] → AUC = 0.85
  Fold 4: 학습=[A,B,C,E] → 검증=[D] → AUC = 0.84
  Fold 5: 학습=[A,B,C,D] → 검증=[E] → AUC = 0.85

  평균 AUC = 0.848 ± 0.007  ← 이것이 진짜 실력
```

```python
# KKBox pipeline_v8.py에서의 실제 코드
skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
# Stratified = "각 폴드에서 재청취/비재청취 비율을 같게 유지"
```

> [!IMPORTANT]
> **KKBox의 가장 뼈아픈 교훈 — OOF 검증 없이는 자기기만이다:**
> 
> | 버전 | 로컬 AUC | Kaggle AUC | 원인 |
> |------|---------|------------|------|
> | v4 (잘못된 TE) | 0.96 | 0.58 | **데이터 누수 → 거짓 성능** |
> | v6a (OOF TE) | 0.85 | 0.67 | 적절한 검증 → 진짜 성능 |
> | v8 (SVD+시퀀스) | 0.85+ | 0.67+ | 피처 품질 개선 |

### 🧭 왜 이것을 배우는가?

"선형회귀, 로지스틱 회귀, 의사결정나무, 랜덤 포레스트, LightGBM 중 뭘 써야 하나?"

또한 정보 이론(Information Theory)은 **"데이터에 얼마나 많은 정보가 들어있는가?"**, **"두 확률분포가 얼마나 다른가?"**를 정량화하여 머신러닝 전반의 토대를 제공한다. 의사결정나무의 분할 기준(정보 이득)이 바로 정보 이론에서 온다.

---

### 6.1 정보 이론의 핵심

#### 📌 용어 해설
- **정보량(information content)**: 어떤 사건이 일어났을 때 얻는 "놀라움의 양". 확률이 낮을수록 많음
- **엔트로피(entropy)**: "평균적인 놀라움" = 불확실성의 척도
- **교차 엔트로피(cross-entropy)**: "실제 분포 P에서 왔는데, 분포 Q를 가정하고 인코딩하면 얼마나 비효율적인가"
- **KL 다이버전스**: 두 분포의 "거리" (정확히는 비대칭)

**정보량:**

```
I(x) = -log₂(P(x))

예시:
  "내일 해가 뜬다" → P = 0.99 → I = -log₂(0.99) = 0.01 bit (놀랍지 않음)
  "유저이 오늘 록 대신 K-POP을 들었다" → P = 0.1 → I = 3.32 bit (놀라움)
  "비 오는 날 검정치마를 안 들었다" → P = 0.01 → I = 6.64 bit (매우 놀라움)
```

**엔트로피:**

```
H(X) = -Σ P(x) × log₂(P(x))

KKBox 유저의 장르 분포에 적용:
  유저A: 락 90%, 팝 5%, 발라드 5% → H = 0.57 bit (취향이 좁음, 예측 쉬움)
  유저B: 락 25%, 팝 25%, 발라드 25%, EDM 25% → H = 2.0 bit (취향이 넓음, 예측 어려움)

→ 엔트로피가 높은 유저가 추천하기 더 어렵다
```

**교차 엔트로피와 KL 다이버전스:**

```
KL(P || Q) = Σ P(x) × log[P(x) / Q(x)]

P = 실제 분포 (유저의 진짜 재청취 패턴)
Q = 모델이 예측한 분포

Cross-Entropy = 엔트로피 + KL 다이버전스
→ Cross-Entropy를 최소화 = KL 다이버전스를 최소화
→ "모델의 예측을 실제에 가깝게 만든다"

Binary Cross-Entropy (Ch.4.2에서 배운 것)가 바로 이것의 이진 버전!
```

## Chapter 7. 트리 기반 모델

> **"대부분의 정형 데이터 문제에서, 잘 튜닝된 GBDT가 딥러닝을 이긴다."**
> — Grinsztajn et al., 2022

### 🧭 왜 이것을 배우는가?

KKBox 대회에서 pipeline v3~v9까지 **모두** LightGBM을 사용했다. 왜?
1. 정형 데이터에서 성능이 최고 수준
2. 학습 속도가 빠름 (수백만 행을 분 단위로 처리)
3. 피처 중요도를 알려줌 (어떤 피처가 도움이 되는지 바로 확인 가능)
4. 범주형/결측값을 자연스럽게 처리

의사결정나무 → 랜덤 포레스트 → Gradient Boosting → LightGBM으로 이어지는 진화 과정을 이해하면,
KKBox 파이프라인의 모든 선택이 **왜** 그렇게 되었는지 알 수 있다.

---

### 7.1 의사결정나무 — "스무고개 게임"

#### 📌 용어 해설
- **의사결정나무(Decision Tree)**: "예/아니오 질문을 반복해서 분류하는 모델". 스무고개와 같은 원리
- **노드(node)**: 질문을 하는 지점. "svd_dot > 0.3인가?"
- **리프(leaf)**: 더 이상 질문하지 않는 끝 지점. 최종 예측값을 출력
- **분할 기준**: "어떤 질문이 가장 유용한가?"를 판단하는 점수

**구체적 예시 — KKBox에서의 의사결정나무:**

```
"이 유저가 이 곡을 다시 들을까?"

질문 1: svd_dot(유저-곡 궁합) > 0.3인가?
├── Yes (40%의 데이터)
│   질문 2: user_artist_count > 5인가?
│   ├── Yes → 예측: 다시 듣는다 (확률 85%) 🎵
│   └── No  → 예측: 반반 (확률 55%)
│
└── No (60%의 데이터)
    질문 3: song_count > 1000인가? (인기곡인가?)
    ├── Yes → 예측: 반반 (확률 45%)
    └── No  → 예측: 안 듣는다 (확률 25%) 🔇
```

**분할 기준 — Gini 불순도 (가장 많이 쓰이는 것):**

```
Gini 불순도 = 1 - Σ(각 클래스 비율)²

예시:
  노드 A: 재청취 50%, 비재청취 50%
    Gini = 1 - (0.5² + 0.5²) = 1 - 0.5 = 0.5 (최대 불순 — 반반이라 못 맞추겠다)

  노드 B: 재청취 90%, 비재청취 10%
    Gini = 1 - (0.9² + 0.1²) = 1 - 0.82 = 0.18 (꽤 순수 — 거의 확실히 재청취)

  노드 C: 재청취 100%, 비재청취 0%
    Gini = 1 - (1.0² + 0.0²) = 0 (완전 순수 — 확신!)

좋은 분할 = 분할 후 자식 노드들의 Gini가 크게 낮아지는 것
```

**정보 이득 — Chapter 6.1의 엔트로피와 연결:**

```
정보 이득(IG) = 부모 엔트로피 - 가중 평균(자식 엔트로피들)

예시:
  부모: 재청취 50%, H = 1.0 bit
  
  분할: svd_dot > 0.3?
    자식1 (40%): 재청취 70%, H = 0.88 bit
    자식2 (60%): 재청취 37%, H = 0.95 bit
  
  IG = 1.0 - (0.4 × 0.88 + 0.6 × 0.95) = 1.0 - 0.922 = 0.078 bit

"svd_dot으로 분할하면 불확실성이 0.078 bit 줄어든다"
→ 이 값이 가장 큰 피처로 첫 분할을 한다
```

**의사결정나무의 장단점:**

```
장점:
  ✅ 해석이 쉽다 (왜 이 예측을 했는지 경로를 따라가면 됨)
  ✅ 비선형 패턴을 자연스럽게 포착
  ✅ 전처리가 거의 필요 없음

단점:
  ❌ 과적합하기 매우 쉬움 (트리를 깊게 키우면 학습 데이터를 외워버림)
  ❌ 하나의 데이터가 바뀌면 전체 트리 구조가 크게 변함 (불안정)
  ❌ 단독으로는 성능이 낮음
```

**→ 이 단점을 해결하기 위해 "여러 트리를 모으는" 앙상블이 등장한다.**

### 7.2 랜덤 포레스트 — "여러 나무의 다수결"

> **이전 연결**: 의사결정나무 하나는 불안정하고 과적합한다.
> **이 절의 해결책**: 많은 나무를 키워서 다수결로 결정하면 훨씬 안정적이다.

#### 📌 용어 해설
- **앙상블(ensemble)**: 여러 모델의 예측을 결합하여 하나의 최종 예측을 만드는 기법
- **배깅(bagging, Bootstrap AGGregatING)**: 데이터를 랜덤하게 복원 추출해서 각 모델에 다른 데이터를 줌
- **부트스트랩(bootstrap)**: "복원 추출로 만든 표본". 같은 데이터가 여러 번 뽑힐 수도 있음

```
랜덤 포레스트 = 100~1000개의 의사결정나무를 만들어 다수결

각 나무의 학습 방식:
  1. 원본 데이터에서 랜덤 복원 추출 (배깅) → 각 나무가 다른 데이터로 학습
  2. 각 분할에서 피처의 일부만 랜덤 선택 → 각 나무가 다른 피처를 봄

예측: 100개 나무 중 73개가 "재청취", 27개가 "비재청취"
→ 최종 예측: 재청취 (73% 투표)
```

**비유:**

```
시험 문제: "이 유저가 이 곡을 다시 들을까?"

한 학생에게만 물어보면? → 맞을 수도, 틀릴 수도 (불안정)

100명의 학생에게 각각 물어보면? (단, 각자 다른 참고자료를 봄)
  → 73명이 "다시 듣는다", 27명이 "안 듣는다"
  → 다수결로 "다시 듣는다"
  → 개인은 틀릴 수 있지만, 다수결의 정확도는 훨씬 높다!

핵심 조건: 학생들이 "서로 다른 방식으로" 틀려야 함.
  만약 모든 학생이 같은 오답을 내면 → 다수결도 틀림
  → 배깅과 피처 랜덤 선택이 "다양성"을 확보하는 역할
```

### 7.3 그래디언트 부스팅 — "첫 번째가 틀린 부분을 두 번째가 보완"

> **이전 연결**: 랜덤 포레스트는 여러 나무를 **독립적으로** 만들어 평균을 냈다.
> **이 절의 새로운 접근**: 나무를 **순서대로** 만들되, 이전 나무의 **실수를 학습**하여 보완한다.
> **다음 연결**: LightGBM은 이 그래디언트 부스팅의 최적화 버전이다.

#### 📌 용어 해설
- **부스팅(boosting)**: 약한 모델을 순차적으로 쌓아 올려 강한 모델을 만드는 기법
- **잔차(residual)**: "정답 - 현재 모델의 예측" = 모델이 틀린 부분
- **의사 잔차(pseudo-residual)**: "손실함수의 그래디언트" = 일반화된 잔차
- **학습률(learning rate, shrinkage)**: 각 트리의 기여를 줄이는 계수. Chapter 1.5의 η와 같은 개념

**배깅 vs 부스팅 — 핵심 차이:**

| | 배깅 (Random Forest) | 부스팅 (Gradient Boosting) |
|---|---|---|
| **나무 학습** | 독립적, 병렬 가능 | **순차적** — 이전 실수를 학습 |
| **목표** | 분산 감소 (안정화) | 편향 감소 (정확도 향상) |
| **과적합 위험** | 적음 | **있음** (트리가 너무 많으면) |

**알고리즘 (비유와 함께):**

```
비유: 연필 초상화를 그리는 과정

1단계 (F₀): 얼굴 윤곽을 대충 그린다 (전체 평균으로 시작)
  → 실수: 눈 위치가 틀리고, 코 크기가 다름

2단계 (F₁): 1단계의 실수(잔차)를 집중적으로 수정
  → 눈 위치를 맞추는 데 집중하는 새 트리
  → F₁ = F₀ + 0.1 × (눈 수정 트리)   ← 0.1 = 학습률

3단계 (F₂): 2단계의 실수를 집중적으로 수정
  → 코 크기를 맞추는 데 집중하는 새 트리
  → F₂ = F₁ + 0.1 × (코 수정 트리)

...반복...

최종: F_2000 = F₀ + 0.1×T₁ + 0.1×T₂ + ... + 0.1×T₂₀₀₀
  → 2000개의 작은 수정이 쌓여 정교한 초상화 완성!
```

**수식:**

```
F₀(x) = 초기 예측값 (전체 데이터의 평균)

반복 (m = 1, 2, ..., M):
  1. 잔차 계산: rᵢ = -(∂L / ∂F_{m-1}(xᵢ))   ← "어디를 고쳐야 하는가"
  2. 잔차를 학습하는 트리 T_m 구축           ← "어떻게 고치는가"
  3. 업데이트: F_m(x) = F_{m-1}(x) + η × T_m(x)  ← "고친 것을 반영"

η: 학습률 — 한 번에 너무 많이 수정하면 과적합
   → KKBox에서 η = 0.1 사용
```

### 7.4 LightGBM — "KKBox의 핵심 엔진"

> **이전 연결**: 그래디언트 부스팅의 원리를 배웠다.
> **이 절의 목적**: LightGBM이 **왜 빠르고**, **어떻게 설정하는지** 실전적으로 이해한다.

#### XGBoost vs LightGBM

```
XGBoost: 좋은 성능, 하지만 대규모 데이터에서 느림
LightGBM: XGBoost와 비슷한 성능 + 훨씬 빠름

속도 비결:
  1. GOSS (그래디언트 기반 샘플링)
     → 학습에 중요한 샘플(예측을 많이 틀린 것) 위주로 학습
     → 전체 다 볼 필요 없이 핵심만 봐서 빠름
  
  2. EFB (배타적 피처 번들링)
     → 동시에 값이 있는 일이 거의 없는 피처들을 하나로 묶음
     → 피처 수 감소 → 빠름
  
  3. Leaf-wise 성장
     → XGBoost: 한 층씩 전부 확장 (level-wise)
     → LightGBM: 손실이 가장 큰 리프부터 확장 → 같은 트리 수에서 더 정확
  
  4. 히스토그램 기반 분할
     → 연속값을 구간(bin)으로 나눠서 분할점 탐색 속도 향상
```

**KKBox v8 하이퍼파라미터 완전 해설:**

```python
params = {
    'objective': 'binary',     # 목표: 이진 분류 (재청취 여부)
    'metric': 'auc',           # 평가 지표: AUC
    'learning_rate': 0.1,      # 학습률: 각 트리의 기여도를 10%로 제한
                               #   → 낮으면 안전하지만 트리 더 필요
                               #   → 높으면 빠르지만 과적합 위험
    
    'num_leaves': 255,         # 한 트리의 최대 리프 수 (= 트리의 복잡도)
                               #   → 클수록 복잡한 패턴 포착 가능
                               #   → 너무 크면 과적합
    
    'max_depth': -1,           # 트리 깊이 제한 없음
                               #   → leaf-wise 성장이라 num_leaves로 제어
    
    'min_child_samples': 100,  # 리프에 최소 100개 샘플
                               #   → 100명 이하의 유저만 해당하는 패턴은 무시
                               #   → 과적합 방지 핵심 파라미터
    
    'subsample': 0.7,          # 각 트리에서 데이터 70%만 랜덤 사용
                               #   → 배깅(Random Forest) 효과 추가
    
    'colsample_bytree': 0.7,   # 각 트리에서 피처 70%만 랜덤 사용
                               #   → 피처 다양성 확보
    
    'reg_alpha': 1.0,          # L1 정규화 (Lasso) — 쓸모없는 분할 억제
    'reg_lambda': 5.0,         # L2 정규화 (Ridge) — 극단적 예측값 억제
    
    'n_estimators': 2000,      # 최대 트리 수 (실제로는 early stopping이 결정)
}
```

> [!TIP]
> **실전 튜닝 순서 (KKBox에서 배운 것):**
> 1. `learning_rate` 고정 (0.05~0.1)
> 2. `n_estimators`를 크게 잡고 `early_stopping`으로 자동 결정
> 3. `num_leaves`, `min_child_samples`로 트리 복잡도 조정
> 4. `subsample`, `colsample_bytree`로 과적합 제어
> 5. `reg_alpha`, `reg_lambda`는 마지막에 미세 조정
> 
> **가장 중요한 교훈:**
> **"하이퍼파라미터 튜닝보다 피처 엔지니어링이 10배 중요하다."**
> v7→v8 개선의 핵심은 SVD 임베딩과 시퀀스 피처 **추가**이지, 파라미터 변경이 아니었다.

## Chapter 8. 피처 엔지니어링 — "모델에 좋은 재료를 넣자"

> **"데이터와 피처가 모델 성능의 상한선을 결정하고,
>   모델과 하이퍼파라미터는 그 상한에 얼마나 가까이 가느냐를 결정한다."**
> — Pedro Domingos

### 🧭 왜 이것을 배우는가?

KKBox 파이프라인을 v3에서 v8까지 개선하면서, **모든 유의미한 AUC 향상은 피처 엔지니어링에서 왔다.** LightGBM 파라미터를 아무리 튜닝해도 0.01~0.02 AUC 차이밖에 안 나지만, **좋은 피처 하나**가 그 이상을 만든다.

**비유:** 최고급 오븐(모델)이 있어도, **재료(피처)**가 나쁘면 맛없는 빵이 나온다. 피처 엔지니어링 = 좋은 재료를 준비하는 과정.

---

### 8.1 KKBox 피처 진화사 v3→v8

```
v3 (기본 메타데이터)        Kaggle ~0.58
  song_length, language, city, registered_via
  → "곡이 3분이고 한국어입니다" 수준의 정보만으로는 한계

v4 (Count 피처)             Kaggle ~0.62  (+4%p)
  user_count, song_count, artist_count
  → "이 유저가 800곡을 들었고, 이 곡은 500명이 들었다"

v5 (비율 피처)              Kaggle ~0.64  (+2%p)
  user_artist_ratio, user_genre_ratio
  → "이 유저의 전체 재생 중 이 아티스트가 12%를 차지한다"

v6 (교차 피처)              Kaggle ~0.67  (+3%p)
  user_song_count, user_artist_count, user_genre_count
  → "이 유저가 이 아티스트를 몇 번, 이 장르를 몇 번 들었는지"

v8 (SVD 임베딩 + 시퀀스)    Kaggle ~0.67+ (+0.5%p)
  svd_dot, same_artist_as_prev
  → "유저-곡의 잠재적 궁합 점수", "직전 곡과 같은 아티스트?"
```

### 8.2 피처 엔지니어링 기법 총정리

#### 핵심 원칙: 트리가 알아서 하는 것 vs 만들어줘야 하는 것

```
❌ 헛수고 — 트리가 알아서 한다:
  구간 나누기: "song_length를 짧음/보통/김으로 나누자"
    → 트리가 알아서 분할점을 찾음

  조건 조합: "a > 5이고 b > 2인 경우"
    → 질문 두 번이면 됨. 트리가 정확히 이걸 함

  단순 변환: "log(song_count)"
    → 트리는 순서만 보니까, log를 씌우나 안 씌우나 분할 결과 동일

✅ 진짜 효과 — 트리가 못 하는 것:
  다른 행 집계:  "이 유저가 총 몇 곡 들었는지" = 다른 행들을 세어야 알 수 있음
  나눗셈(비율):  user_artist_count / user_count = 트리는 나눗셈을 못 함
  행렬 분해:     svd_dot = 수학적 변환으로만 얻을 수 있는 값
  시퀀스 참조:   "직전 곡과 같은 아티스트?" = 다른 행(직전 행)을 봐야 함
```

```
트리가 한 행(row)을 볼 때:
  (유저A, 곡X, song_length=3.8, genre=록, ...)
  
  → 이 행의 값들만 보고 질문을 던진다
  → "유저A가 다른 곡을 몇 개 들었는지"는 다른 행에 있음 → 못 봄!
  → 그래서 미리 세어서(count) 이 행에 붙여줘야 한다!
```

**1) Count/빈도 기반 — "다른 행을 세어서 현재 행에 붙이기":**

```python
# "이 유저는 얼마나 활발한가?" → 활발한 유저일수록 재청취 확률 ↑
user_count = train['msno_code'].value_counts()

# "이 곡은 얼마나 인기 있는가?" → 인기곡은 재청취 확률 ↑
song_count = train['song_id_code'].value_counts()

# "이 유저가 이 아티스트를 얼마나 좋아하는가?" → 핵심 시그널!
user_artist_count = train.groupby(['msno_code', 'artist_code']).size()
```

```
구체적으로:
  원본 행: (유저A, 곡X)  ← 이 행만 봐서는 유저A의 활발도를 모름
  
  다른 행들을 세면: 유저A가 전체에서 8502번 등장
  → user_count = 8502를 이 행에 붙임
  → 트리가 "user_count > 1000?" 같은 질문 가능!
```

**2) 비율 변환 — "나눗셈은 트리가 못 하니까 미리 계산":**

```python
# 절대 횟수보다 상대 비율이 더 유용
user_artist_ratio = user_artist_count / (user_count + 1)
# 의미: "이 유저의 전체 재생 중 이 아티스트의 비율"
# 유저A: 8000곡 중 잔나비 400곡 = 5% (다양한 취향에서 잔나비도 좋아함)
# 유저B: 100곡 중 잔나비 40곡 = 40% (거의 잔나비만 들음)
# → 같은 "40곡"이지만 의미가 완전히 다름!
```

```
왜 이걸 트리가 못 하는가:
  유저A: user_artist_count=400, user_count=8000 → 비율 5%
  유저B: user_artist_count=40,  user_count=100  → 비율 40%
  
  트리의 질문: "user_artist_count > 100?"
    유저A: 400 > 100 → 예
    유저B: 40 > 100  → 아니오
    → 비율이 높은 유저B가 "아니오" 쪽으로! 엉뚱한 분류 ❌
  
  비율 피처를 넣으면:
    "user_artist_ratio > 0.1?"
    유저A: 0.05 → 아니오 (잔나비 비중 낮음)
    유저B: 0.40 → 예 (잔나비 비중 높음) ✅
```

**3) Target Encoding — "양날의 검":**

**Target Encoding이란?**

```
범주형 데이터(글자)를 숫자로 바꾸는 방법 중 하나.
핵심: 그 범주의 "정답(target) 평균"으로 바꾼다.

  "잔나비"의 데이터 500건 중 375건이 재청취
  → 잔나비 = 375/500 = 0.75

  "BTS"의 데이터 1000건 중 600건이 재청취
  → BTS = 600/1000 = 0.60

  "아이유"의 데이터 800건 중 656건이 재청취
  → 아이유 = 656/800 = 0.82
```

**다른 인코딩과 비교:**

```
  Label Encoding: 잔나비=1, BTS=2, 아이유=3
    → 숫자의 크기에 의미 없음: 아이유(3) > BTS(2)? 의미 없다 ❌

  One-Hot Encoding: 잔나비=[1,0,0], BTS=[0,1,0], 아이유=[0,0,1]
    → 의미는 있지만, 아티스트 10만 명이면 피처 10만 개 → 비현실적 ❌

  Target Encoding: 잔나비=0.75, BTS=0.60, 아이유=0.82
    → 크기에 의미 있음! (재청취율 높은 아티스트일수록 숫자가 크다) ✅
    → 피처 1개로 끝 ✅
```

> [!CAUTION]
> **KKBox v4에서 가장 심각한 실수: 데이터 누수의 재앙**
>
> KKBox에서 user_id, song_id, artist, genre, composer, lyricist 등
> 모든 범주형 피처에 Target Encoding을 적용했는데, 전부 누수가 발생.
>
> ```python
> # ❌ 잘못된 방법: 전체 train의 target 평균을 피처로 사용
> artist_mean = train.groupby('artist_code')['target'].mean()
> train['artist_te'] = train['artist_code'].map(artist_mean)
> # → 학습할 때 "정답"을 이미 알고 있는 셈 = 컨닝!
> # → 로컬 AUC 0.96 (시험지의 답을 외운 것)
> # → Kaggle AUC 0.58 (새 시험지에서 탈탈 털림)
> ```

**왜 누수가 발생하는가 — 건수가 적을수록 심각:**

```
아티스트 "잔나비": 500건 → 평균 0.75
  행37 (정답=1): artist_te = 0.75  ← 자기 정답(1)이 1/500만큼 반영
  → 영향 작음. 하지만 여전히 누수는 존재

유저Y: 딱 1건 (정답=1)
  행5: user_te = 1/1 = 1.0  ← 자기 정답 그 자체!
  → "user_te = 1.0이면 정답은 반드시 1" = 답지 복사!

유저Z: 딱 1건 (정답=0)
  행6: user_te = 0/1 = 0.0
  → "user_te = 0.0이면 정답은 반드시 0" = 또 답지!

  → 트리가 "TE 높으면 1, 낮으면 0" 하면 거의 정답을 맞춤
  → 학습이 아니라 답지 외우기!
```

> ```python
> # ✅ 올바른 방법: OOF (Out-of-Fold) Target Encoding
> # 자신의 정답을 절대 참조하지 않도록, "다른 폴드"의 데이터만으로 계산
> for fold in folds:
>     train_part, val_part = fold
>     # val_part의 TE는 train_part만으로 계산
>     global_mean = train_part['target'].mean()
>     artist_stats = train_part.groupby('artist_code')['target'].agg(['mean', 'count'])
>     # Smoothing: 샘플이 적을수록 전체 평균에 가깝게 당김
>     smoothed = (artist_stats['mean'] * artist_stats['count'] + global_mean * α) / (artist_stats['count'] + α)
> ```

**OOF가 해결하는 원리:**

```
데이터를 5등분. 유저Y(1건)가 Fold1에 있다면:

  유저Y의 Fold1 TE = Fold2~5에서 유저Y의 데이터로 계산
  → 유저Y가 Fold2~5에 없으면? → 전체 평균(0.50)을 사용
  
  행5: user_te = 0.50 (자기 정답 1.0이 아님!)
  → 모델은 이 값만 보고는 정답을 알 수 없음 ✅
  
  결과: 로컬 AUC = 0.67, Kaggle AUC = 0.67 (일치!) ✅
```

**4) 임베딩 기반 피처 (SVD):**

```
전통적 피처: 사람이 직접 설계 (user_count, user_artist_ratio, ...)
SVD 임베딩: 데이터에서 자동으로 "잠재적 취향"을 발견

SVD 임베딩 피처의 해석 (실제로는 모델이 자동 결정):
  user_svd_0 ≈ "인디록 vs K-POP" 선호도 축
  user_svd_1 ≈ "발라드 vs 댄스" 선호도 축
  ...
  svd_dot = 유저 벡터 · 곡 벡터 = 종합 궁합 점수
```

**5) 시퀀스/컨텍스트 피처:**

```python
# KKBox v8 — "직전 곡과의 관계"
train['same_artist_as_prev'] = (
    train['artist_code'] == train.groupby('msno_code')['artist_code'].shift(1)
)
# "직전에 들은 곡과 같은 아티스트인가?"
# → 같은 아티스트를 연속으로 들으면 재청취 확률 UP (몰입 상태)
# → 음악 프로젝트의 "Tier S 아티스트" 개념과 직결!
```

## Chapter 9. 앙상블 방법론

### 🧭 왜 이것을 배우는가?

KKBox pipeline_v8.py에서 최종 예측은 이렇게 만들어진다:
```python
test_preds += model.predict_proba(X_test)[:, 1] / N_FOLDS  # 5개 모델의 평균
```

이것이 앙상블이다. "왜 하나의 모델 대신 여러 모델의 평균을 내는가?" — 이 질문에 대한 이론적 답이 이 챕터에 있다.

---

### 9.1 앙상블의 이론적 근거 — "다수결의 힘"

```
콘도르세의 배심원 정리:
  각 배심원이 올바른 판단을 내릴 확률이 50%보다 높고,
  배심원들이 서로 독립적으로 판단하면:
  → 배심원 수가 많아질수록, 다수결의 정확도는 100%에 수렴한다.

핵심 조건: 다양성 (Diversity)
  만약 모든 배심원이 같은 편견으로 틀리면 → 다수결도 틀림
  서로 다른 관점에서 다른 실수를 해야 → 다수결로 상쇄됨
```

**"다르게 틀려야 의미 있다":**

```
모델 A: [맞, 맞, 틀, 맞, 틀]
모델 B: [맞, 틀, 맞, 맞, 틀]
평균:    [맞, 맞, 맞, 맞, 틀]  ← 서로 다른 곳에서 틀려서 보완됨! ✅

모델 A: [맞, 맞, 틀, 맞, 틀]
모델 C: [맞, 맞, 틀, 맞, 틀]  ← A랑 똑같이 틀림
평균:    [맞, 맞, 틀, 맞, 틀]  ← 합쳐봤자 달라진 게 없음 ❌
```

---

### 9.2 앙상블의 4가지 레벨

**Level 1: 같은 모델, 다른 데이터 (K-Fold 평균)**

```
→ LightGBM 5개를 다른 80% 데이터로 학습 → 예측 평균
→ KKBox에서 한 것
→ 언제나 해야 함. 안 할 이유가 없음 ✅

  Fold1 모델 → 예측: 0.72
  Fold2 모델 → 예측: 0.68
  Fold3 모델 → 예측: 0.74
  Fold4 모델 → 예측: 0.70
  Fold5 모델 → 예측: 0.71
  최종 = 평균 = 0.71
```

**Level 2: 같은 종류, 다른 설정**

```
→ LightGBM (num_leaves=127) + LightGBM (num_leaves=255) 평균
→ 비슷한 모델이라 예측도 비슷 → 효과 작음 ❌
→ 보통 안 함
```

**Level 3: 다른 종류 모델 섞기 (Blending)**

```
→ LightGBM × 0.7 + 로지스틱 회귀 × 0.3 → 가중 평균
→ Titanic에서 한 것!

  LightGBM 예측: 0.85 (비선형 패턴에 강함)
  로지스틱 회귀:  0.70 (선형 관계에 안정적)
  가중 평균: 0.85×0.7 + 0.70×0.3 = 0.805

  → 두 모델이 "다른 방식으로 틀리니까" 보완 효과 ✅
```

**Level 4: 스태킹 (Stacking) — 가장 강력**

```
→ 1층: LightGBM, XGBoost, 로지스틱 회귀 각각 예측
→ 2층: 세 예측을 "피처"로 넣어서 새 모델이 최종 결정

  원본 피처 → LightGBM  → 예측 0.85 ─┐
  원본 피처 → XGBoost   → 예측 0.80 ─┼→ [0.85, 0.80, 0.70] → 2층 모델 → 최종 0.83
  원본 피처 → 로지스틱  → 예측 0.70 ─┘

  → 가중치를 사람이 정하지 않고 2층 모델이 자동으로 학습!
  → Kaggle 상위권의 필수 기법
```

---

### 9.3 언제 다른 모델을 섞는가?

```
판단 기준: 두 모델의 예측값 상관계수

  상관 0.95+: 거의 같은 예측 → 섞어봤자 효과 작음
  상관 0.7~0.9: 적당히 다름 → 섞으면 효과 있음 ✅
  상관 0.5 이하: 너무 다름 → 하나가 크게 잘못된 것일 수 있음

왜 트리 + 선형 모델을 섞으면 좋은가:
  트리(LightGBM)가 잘하는 것:
    → 비선형 패턴, 피처 간 상호작용
    → "svd_dot > 1.5이면서 genre=록이면 재청취 높음"

  선형 모델(로지스틱)이 잘하는 것:
    → 단순한 선형 관계, 극단값에 안정적
    → "user_count가 높으면 대체로 재청취 높음"

  → 다른 방식으로 틀리니까 보완됨!
```

**실전 가이드:**

```
시간 없을 때:      5-Fold 평균만 (Level 1) → 이것만으로도 충분
시간 좀 있을 때:   LightGBM + XGBoost 평균 (Level 3) → 쉽고 효과적
Kaggle 상위 노릴 때: 스태킹 (Level 4) → 복잡하지만 +0.005 가능
```

# Part 4: 딥러닝

> **이 Part를 배우는 이유:**
> 
> Part 3의 LightGBM은 정형 데이터에서 최강이다. 하지만 한계가 있다:
> - **피처를 사람이 직접** 만들어야 한다 (SVD, same_artist_as_prev 등)
> - **시퀀스(순서) 데이터**를 자연스럽게 처리하지 못한다
> - **비정형 데이터**(이미지, 오디오, 텍스트)를 다루지 못한다
> 
> 딥러닝은 이 한계를 극복한다:
> - **피처를 자동으로 학습**한다 (피처 엔지니어링 ≈ 불필요)
> - **시퀀스를 자연스럽게** 처리한다 (RNN, Transformer)
> - **음악 오디오 자체**에서 특성을 추출할 수 있다 (CNN)
> 
> **KKBox 다음 스텝**: LightGBM + 수동 피처 → 딥러닝 기반 추천 모델로 진화

## Chapter 10. 신경망 기초

> **"신경망은 범용 함수 근사기다."**
> — 📘 DLBook Ch.6

### 🧭 왜 이것을 배우는가?

Chapter 4의 로지스틱 회귀는 "하나의 뉴런"과 동일하다. 이것을 여러 층으로 쌓으면 **신경망**이 된다.
신경망이 로지스틱 회귀와 다른 점은 딱 하나: **비선형 패턴을 자동으로 포착**할 수 있다는 것.

---

### 10.1 퍼셉트론에서 다층 신경망으로

#### 📌 용어 해설
- **뉴런(neuron)**: 신경망의 기본 단위. "입력을 받아 계산하고 출력하는 작은 상자"
- **층(layer)**: 뉴런을 가로로 나열한 것
- **은닉층(hidden layer)**: 입력과 출력 사이에 있는 층. "숨겨진 연산"을 수행
- **활성함수(activation function)**: 뉴런의 출력에 비선형성을 추가하는 함수. 이것이 없으면 아무리 층을 쌓아도 선형회귀와 동일

```
단일 뉴런 (= 로지스틱 회귀와 동일):
  입력: [x₁=0.12, x₂=342, x₃=2.34]  (user_artist_ratio, song_count, svd_dot)
  
  1단계 (선형 결합): z = w₁×x₁ + w₂×x₂ + w₃×x₃ + b
                       = 0.5×0.12 + 0.01×342 + 0.3×2.34 + (-0.5)
                       = 3.68
  
  2단계 (활성함수): a = σ(z) = σ(3.68) = 0.975
  
  출력: 0.975 (= 재청취 확률 97.5%)

다층 신경망 (MLP):
  입력층:  [x₁, x₂, x₃, ..., x₁₀₀]          ← 100개 KKBox 피처
    ↓
  은닉층1: 256개 뉴런 (각각 "선형결합 + 활성함수")
    ↓
  은닉층2: 128개 뉴런
    ↓
  은닉층3: 64개 뉴런
    ↓
  출력층:  1개 뉴런 → σ(z) → 재청취 확률
```

**왜 층을 쌓는가? — 범용 근사 정리(Universal Approximation Theorem):**

```
이론: 충분히 넓은 1개의 은닉층만으로도, 어떤 연속함수든 근사 가능.

현실: 하지만 1개 층으로 하려면 뉴런이 천문학적으로 많이 필요.
      여러 층을 쌓으면 훨씬 적은 뉴런으로 같은 복잡도 표현 가능.
      
비유:
  1층 건물로 도서관을 만들 수 있지만, 축구장만한 면적이 필요.
  5층 건물로 만들면 훨씬 작은 부지에 같은 도서관 건설 가능.
```

---

### 10.2 활성함수 — "비선형의 마법"

```
만약 활성함수가 없다면?
  은닉층1: h₁ = W₁x + b₁
  은닉층2: h₂ = W₂h₁ + b₂ = W₂(W₁x + b₁) + b₂ = W₂W₁x + (W₂b₁ + b₂)
  → 결국 y = Ax + B (그냥 선형 변환!)
  → 아무리 층을 쌓아도 선형회귀와 동일

활성함수를 넣으면:
  h₁ = 활성함수(W₁x + b₁)
  h₂ = 활성함수(W₂h₁ + b₂)
  → 비선형이 겹겹이 더해져 → 복잡한 패턴 포착 가능!
```

**주요 활성함수:**

| 함수 | 수식 | 출력 범위 | 특징 |
|------|------|----------|------|
| **Sigmoid** | 1/(1+e⁻ᶻ) | (0, 1) | 확률 해석 가능. 깊은 층에서 기울기가 사라지는 문제 |
| **ReLU** | max(0, z) | [0, ∞) | 가장 널리 사용. 빠르고 단순. 음수면 0 출력 |
| **Leaky ReLU** | max(0.01z, z) | (-∞, ∞) | ReLU의 "죽은 뉴런" 문제 해결 |
| **GELU** | z × Φ(z) | (-∞, ∞) | Transformer에서 표준. 부드러운 활성화 |

---

### 10.3 역전파 — "오차를 뒤로 보내 모든 층을 학습시키기"

#### 📌 용어 해설
- **순전파(forward pass)**: 입력 → 출력으로 계산하는 과정 (예측)
- **역전파(backpropagation)**: 출력 → 입력 방향으로 기울기를 전파하는 과정 (학습)
- **연쇄 법칙(chain rule)**: 합성함수의 미분 = 각 단계의 미분을 곱하는 것

```
순전파 (왼→오): 
  입력 → 은닉층1 → 은닉층2 → 출력 → 손실(Loss) 계산

역전파 (오→왼):
  Loss → "출력층 가중치를 어떻게 바꿔야 하나?"
       → "은닉층2 가중치를 어떻게 바꿔야 하나?"
       → "은닉층1 가중치를 어떻게 바꿔야 하나?"

비유: 
  요리 결과(맛)이 나빴을 때,
  "마지막에 넣은 소금이 문제인가?" → "그 전에 볶는 시간이 문제인가?" → "재료 손질이 문제인가?"
  역순으로 원인을 추적해서 각 단계의 담당자에게 "이렇게 수정해라"고 알려주는 것.
```

**기울기 소실 문제 (Vanishing Gradients):**

```
Sigmoid의 최대 미분값 = 0.25
10개 층을 거치면: 0.25¹⁰ = 0.0000009536...
→ 앞쪽 층의 그래디언트가 거의 0 → 학습이 안 됨!

해결책:
  1. ReLU 활성함수 (미분 = 0 또는 1, 곱해도 안 줄어듦)
  2. Residual Connection (Ch.11의 ResNet) — 그래디언트의 "지름길"
  3. Batch Normalization — 각 층의 입력을 정규화
  4. LSTM의 게이트 (Ch.12) — 셀 상태의 기울기 보존
```

---

### 10.4 최적화 알고리즘 — "경사하강법의 업그레이드"

> **이전 연결**: Ch.1.5에서 기본 경사하강법을 배웠다.
> **이 절의 목적**: 딥러닝에서 실제로 사용하는 개선된 최적화 알고리즘을 배운다.

```
SGD (확률적 경사하강법):
  전체 데이터 → 미니배치(64~256개)로 잘라서 학습
  → 근사적 그래디언트지만 훨씬 빠름

SGD + Momentum (관성):
  이전 업데이트 방향을 기억 → 관성처럼 계속 같은 방향으로 이동
  → 지역 최소점(local minimum)에 빠져도 빠져나올 수 있음

Adam (거의 모든 딥러닝의 표준):
  Momentum + 각 파라미터별 학습률 자동 조정
  → "자주 업데이트되는 파라미터는 학습률 ↓, 드문 파라미터는 학습률 ↑"
  → 추천시스템에서 특히 유용 (인기곡은 자주, 비인기곡은 드물게 등장)
```

---

### 10.5 딥러닝의 정규화 기법

| 기법 | 원리 | 비유 |
|------|------|------|
| **Dropout** | 학습 시 뉴런을 랜덤으로 꺼버림 (보통 50%) | "팀 프로젝트에서 매번 다른 멤버가 빠져도 잘 돌아가게 만들기" |
| **Batch Normalization** | 각 층의 입력을 정규화 | "각 부서에 들어오는 서류 양식을 통일하기" |
| **Weight Decay** | L2 정규화의 딥러닝 버전 | LightGBM의 reg_lambda와 같은 개념 |
| **Early Stopping** | 검증 손실이 올라가면 학습 중단 | KKBox에서도 `early_stopping(50)` 사용 |

## Chapter 11. 합성곱 신경망 (CNN) — "이미지와 오디오를 이해하는 모델"

> 📕 참조: CS231n

### 🧭 왜 이것을 배우는가?

직접적으로 KKBox에서 쓰이지는 않지만, **음악 추천의 미래**와 관련이 깊다:
- 스포티파이는 CNN으로 **오디오 자체의 특성**을 추출해서 추천에 활용한다
- 음악 프로젝트의 37차원 오디오 피처 엔진(MFCC, 템포, 에너지)은 CNN이 **자동으로 하는 일**의 수동 버전이다

---

### 11.1 합성곱 연산

#### 📌 용어 해설
- **합성곱(convolution)**: 작은 필터(커널)를 입력 위에 슬라이딩하면서 "패턴 찾기"를 반복하는 연산
- **커널/필터(kernel/filter)**: 패턴을 감지하는 작은 가중치 행렬. 학습을 통해 자동으로 결정
- **특성 맵(feature map)**: 필터를 적용한 결과. "이 위치에 이 패턴이 얼마나 있는가"

```
비유: 돋보기로 사진을 훑어보기

  원본 이미지                              특성 맵
  ┌─────────────────┐                    ┌──────────┐
  │ 🏔️ 산 사진      │   3×3 필터         │ 여기에    │
  │                 │ ─[가로선감지]→      │ 수평선    │
  │ ═══════════     │                    │ 발견!    │
  │                 │                    │          │
  └─────────────────┘                    └──────────┘

핵심 장점:
  파라미터 공유: 같은 필터로 이미지 전체를 스캔 → 파라미터 수 대폭 감소
  이동 불변성: 패턴이 어디에 있든 같은 필터로 감지 가능
```

**음악에서의 CNN — 스포티파이의 접근:**

```
오디오 파형 → 멜 스펙트로그램 변환 (시간 × 주파수 → 2D "이미지")

  시간 →
  ┌──────────────────┐  ↑
  │ ▓░▓░▓░░░░░░░░░░░ │  주
  │ ░▓░▓░▓░░░░░░░░░░ │  파
  │ ░░░░░░▓▓▓▓░░░░░░ │  수
  │ ░░░░░░░░░░▓▓▓▓▓▓ │  
  └──────────────────┘  ↓

CNN으로 이 "그림"에서:
  - "드럼 비트 패턴" 자동 감지 → 리듬 유형 분류
  - "기타 코드 진행" 자동 감지 → 장르 분류
  - "보컬 질감" 자동 감지 → 유사한 목소리의 가수 매칭

→ 37차원 수동 피처 엔진 대신 CNN이 자동으로 수백 개의 특성을 추출
```

## Chapter 12. 순환 신경망 (RNN) — "시퀀스(순서)를 이해하는 모델"

> 📕 참조: 📘 DLBook Ch.10, CS224n

### 🧭 왜 이것을 배우는가?

음악 재생 이력은 **시퀀스(순서가 있는 데이터)**다:

```
유저의 최근 재생 이력:
  [잔나비 '따윈' → 뉴진스 'Ditto' → 검정치마 '나랑' → 잔나비 '주저하는' → ...]
```

이 **순서** 속에 패턴이 있다 (같은 아티스트를 연속으로 들으면 몰입 상태, 등). KKBox v8의 `same_artist_as_prev` 피처는 이것의 **1단계 수동 구현**이고, RNN은 이것을 **자동으로 학습**한다.

---

### 12.1 기본 RNN

```
시점 t에서:
  h_t = 활성함수(W_hh × h_{t-1} + W_xh × x_t + b)

  h_t: 현재 시점의 은닉 상태 (= "여태까지 들은 곡들의 요약")
  h_{t-1}: 직전 시점의 은닉 상태 (= "바로 전까지 들은 곡들의 요약")
  x_t: 현재 입력 (= "지금 이 곡의 특성 벡터")

핵심: 이전 시점의 "기억" h_{t-1}이 현재로 전달되는 것!
```

**비유:**

```
책을 읽을 때:
  지금 읽는 문장(x_t)만으로는 의미를 이해하기 어려움
  "지금까지 읽은 내용의 요약(h_{t-1})" + "현재 문장(x_t)"을 결합해야 이해 가능
  
  마찬가지로:
  "지금 듣는 곡"만으로 다음에 뭘 추천할지 결정 불가
  "지금까지 들은 곡들의 패턴" + "지금 곡"을 결합해야 적절한 추천 가능
```

**문제: 긴 시퀀스에서 "앞부분 기억"이 사라진다**

```
100곡을 들었다면:
  곡100의 은닉 상태 h₁₀₀에 곡1의 정보가 남아있을까?
  → 99번의 행렬 곱을 거치면서 정보가 거의 사라짐 (기울기 소실과 같은 원리)
  → 해결: LSTM
```

---

### 12.2 LSTM — "무엇을 기억하고 무엇을 잊을지 스스로 결정"

#### 📌 용어 해설
- **LSTM (Long Short-Term Memory)**: "장/단기 기억"을 모두 유지하는 RNN의 발전형
- **게이트(gate)**: "정보를 얼마나 통과시킬지" 결정하는 밸브 (0=차단, 1=전부 통과)
- **셀 상태(cell state)**: 장기 기억을 저장하는 "컨베이어 벨트"

```
LSTM의 3개 게이트:

  Forget Gate (잊기 문):
    "이전 기억 중 무엇을 잊을까?"
    f_t = σ(W_f × [h_{t-1}, x_t] + b_f)
    → 0에 가까우면: 이전 기억 삭제
    → 1에 가까우면: 이전 기억 유지

  Input Gate (입력 문):
    "새로운 정보 중 무엇을 기억할까?"
    i_t = σ(W_i × [h_{t-1}, x_t] + b_i)
    → 0에 가까우면: 새 정보 무시
    → 1에 가까우면: 새 정보 저장

  Output Gate (출력 문):
    "기억 중 무엇을 출력할까?"
    o_t = σ(W_o × [h_{t-1}, x_t] + b_o)
```

**음악 재생 이력에서의 LSTM 동작 비유:**

```
유저의 재생 시퀀스: [잔나비 → 뉴진스 → 검정치마 → 잔나비 → ...]

Forget Gate: "뉴진스는 평소 록 취향과 다르니, 이 기억은 약하게" (f ≈ 0.2)
Input Gate:  "검정치마를 연속으로 듣기 시작했으니, 이건 강하게 기억" (i ≈ 0.9)
Cell State:  "현재 분위기 = 한국 인디록 몰입 중"
Output Gate: "다음 곡 추천에는 '인디록 몰입' 정보를 활용" (o ≈ 0.8)

→ LSTM이 자동으로 "지금 유저가 어떤 분위기인지"를 추적
→ KKBox v8의 same_artist_as_prev를 수동으로 만드는 대신, LSTM이 자동으로 포착
```

## Chapter 13. 어텐션과 트랜스포머 — "필요한 곳에만 집중하기"

> **"Attention Is All You Need"** — Vaswani et al., 2017
> 📕 참조: CS224n Lectures 8-10

### 🧭 왜 이것을 배우는가?

RNN/LSTM의 한계: 시퀀스를 처음부터 끝까지 **순서대로** 처리해야 한다 → 느리고, 먼 과거의 정보가 희석.

Transformer는 이 문제를 해결한다: 시퀀스의 **모든 위치를 동시에** 보면서, **관련 있는 위치에만 집중**한다.

현재 음악 추천 최첨단 모델들(SASRec, BERT4Rec)은 모두 Transformer 기반이다.

---

### 13.1 어텐션의 직관

```
RNN의 방식:
  유저의 100곡 이력: [곡1, 곡2, ..., 곡99, 곡100]
  곡1 → 곡2 → ... → 곡100 순서로 처리
  → 곡1의 정보가 곡100에 도달할 때쯤 거의 사라짐

어텐션의 방식:
  "지금 추천할 곡과 관련 있는 곡은 과거의 어떤 곡인가?"
  → 곡47이 가장 관련 있으면, 직접 곡47에 높은 가중치를 부여
  → 99단계를 거칠 필요 없이 "직빵 연결"!

비유: 도서관에서 책 찾기
  RNN = 1번 서가부터 순서대로 훑기 (느림)
  어텐션 = 목록(인덱스)을 보고 원하는 책에 바로 접근 (빠름)
```

### 13.2 Scaled Dot-Product Attention

```
Attention(Q, K, V) = softmax(Q × Kᵀ / √d_k) × V

Q (Query, 질의):  "내가 찾고 있는 것" — 지금 추천하려는 곡의 특성
K (Key, 키):      "나를 설명하는 것" — 과거 재생 곡들의 특성  
V (Value, 값):    "나의 정보" — 과거 재생 곡들이 전달할 정보
√d_k:             스케일링 (내적이 너무 커서 softmax가 극단적이 되는 것 방지)
```

**구체적 예시:**

```
현재 쿼리(Q): "인디록, 한국어, 느린 템포" (추천하려는 곡의 방향)

과거 재생 이력 (Key들):
  곡7:  "EDM, 영어, 빠른 템포"     → Q·K = 0.1 (관련 낮음) → softmax → 2%
  곡23: "인디록, 한국어, 보통 템포"  → Q·K = 0.8 (관련 높음) → softmax → 45%
  곡47: "인디록, 한국어, 느린 템포"  → Q·K = 0.95 (매우 관련) → softmax → 50%
  곡89: "발라드, 한국어, 느린 템포"  → Q·K = 0.3 (약간 관련) → softmax → 3%

최종 출력 = 2% × 곡7정보 + 45% × 곡23정보 + 50% × 곡47정보 + 3% × 곡89정보
→ 곡47과 곡23의 정보가 압도적으로 반영됨
```

> [!NOTE]
> KKBox v8의 `svd_dot = user_vec · song_vec`는 사실상
> **어텐션 점수의 가장 단순한 버전**이다.
> 어텐션은 이것을 "모든 과거 곡들에 대해 동시에" 계산한다.

### 13.3 Multi-Head Attention — "여러 관점에서 동시에 집중"

```
Head 1: "장르가 비슷한 곡에 집중" → 인디록 곡들에 높은 가중치
Head 2: "같은 아티스트의 곡에 집중" → 잔나비 곡들에 높은 가중치
Head 3: "비슷한 시간대에 들었던 곡에 집중" → 밤에 들은 곡들에 높은 가중치
Head 4: "비슷한 길이의 곡에 집중" → 3~4분 곡에 높은 가중치

→ 4가지 관점의 결과를 합쳐서 → 종합적인 추천 생성
```

## Chapter 14. 추천 시스템 — "KKBox 대회와 음악 프로젝트의 최종 목표"

> 📕 참조: Koren et al. 2009 (Netflix Prize), He et al. 2017 (NCF)

### 🧭 왜 이것을 배우는가?

지금까지 배운 모든 것이 여기서 합쳐진다:
- **선형대수** → 행렬 분해 (협업 필터링)
- **통계** → 확률 추정, 가설검정
- **기계학습** → LightGBM (KKBox)
- **딥러닝** → NCF, Transformer 기반 추천

이 Chapter는 "KKBox 대회에서 실제로 한 일"과 "앞으로 할 수 있는 일"의 전체 지도다.

---

### 14.1 추천 시스템의 세 가지 패러다임

```
1. 협업 필터링 (CF: Collaborative Filtering)
   "비슷한 유저들이 좋아하는 것을 추천"
   → 콘텐츠 무관, 행동 데이터만 활용
   → KKBox의 SVD 임베딩이 이 접근

2. 콘텐츠 기반 (Content-Based)
   "이 곡과 비슷한 특성의 곡을 추천"
   → 장르, 아티스트, 오디오 특성 활용
   → 음악 프로젝트의 태그 벡터, MFCC 피처

3. 하이브리드 (Hybrid)
   CF + 콘텐츠 결합
   → KKBox v8이 사실상 하이브리드:
     SVD (CF) + song_length, artist, language (콘텐츠)
```

### 14.2 행렬 분해 기반 협업 필터링 — 자세히 보기

```
Netflix Prize(2009) 우승의 핵심:

  예측 = 전체 평균(μ) + 유저 편향(b_u) + 아이템 편향(b_i) + 유저 벡터·아이템 벡터(u·v)

  μ = 3.5 (전체 평균 평점)
  b_u = +0.3 (이 유저는 평균보다 후한 평가)
  b_i = -0.2 (이 영화는 평균보다 낮은 평가)
  u·v = +0.8 (이 유저와 이 영화의 궁합이 좋음)
  
  예측 평점 = 3.5 + 0.3 - 0.2 + 0.8 = 4.4점

KKBox에서의 동일한 구조:
  svd_dot = user_factors[유저] · song_factors[곡]  ← 이것이 u·v!
  user_count = 유저 활동량                          ← 유저 편향에 해당
  song_count = 곡 인기도                            ← 아이템 편향에 해당
```

### 14.3 암묵적 피드백의 도전

> [!IMPORTANT]
> **KKBox는 "명시적 평점"이 아닌 "암묵적 상호작용" 데이터를 다룬다.**

```
명시적 피드백: "이 곡 별 5개!" (Netflix 평점) → 선호도 명확

암묵적 피드백: "이 곡을 들었다" (KKBox) → 선호도 불명확
  들었다 = 좋아서? 셔플에 나와서? 실수로?
  안 들었다 = 싫어서? 모르는 곡이라서? 질려서?
```

**음악 프로젝트의 핵심 발견과 연결:**

```
기존 플랫폼의 한계: "안 들음 = 싫음"으로 단순 처리

하지만 실제로는 세 가지 원인이 있다:
  1. 모르는 곡 → 노출 안 됐을 뿐, 좋아할 수 있음 (추천 기회!)
  2. 알지만 취향 아님 → 진짜 부정적 신호
  3. 알고 좋아하지만 질림 → 음원 생애주기의 "Frozen" 상태

→ 음악 프로젝트의 "2-Layer 비대칭 생애주기 모델"은 바로 3번을 감지하기 위한 것
→ Zone 1~4 분류 (전반/후반 스킵률 비교)가 이 "질림"을 정량화
```

### 14.4 딥러닝 기반 추천 모델

**NCF (Neural Collaborative Filtering, 2017):**

```
┌─────────────────────────────────────┐
│         Output Layer (σ)            │  → 재청취 확률
├──────────┬──────────────────────────┤
│ GMF Path │     MLP Path             │
│ u ⊙ v    │ [u, v] → h₁ → h₂ → h₃  │
├──────────┼──────────────────────────┤
│ User Emb │   Item Emb               │  ← SVD와 유사한 임베딩
└──────────┴──────────────────────────┘

GMF = 행렬 분해의 신경망 버전 (선형적 궁합)
MLP = 비선형적 유저-곡 상호작용을 학습
둘을 합치면 → SVD보다 풍부한 패턴 포착 가능
```

**Wide & Deep (Google, 2016):**

```
Wide 부분: 교차 피처의 선형 결합 → "이 유저는 이 특정 곡을 좋아한다" (암기)
Deep 부분: 임베딩 → MLP          → "이런 유형의 유저는 이런 유형의 곡을 좋아한다" (일반화)

KKBox v8과의 정확한 대응:
  Wide ≈ LightGBM의 count/비율 피처 (user_artist_count, song_count 등)
  Deep ≈ SVD 임베딩 피처 (user_svd_*, song_svd_*, svd_dot)
  
→ KKBox v8은 Wide & Deep의 "트리 기반 버전"!
```

**SASRec / BERT4Rec (Transformer 기반):**

```
유저의 최근 재생 시퀀스를 Transformer에 입력
→ Self-Attention으로 시퀀스 내 모든 곡 간의 관계를 동시에 학습
→ 마지막 위치의 출력 = "다음에 이 유저가 들을 곡의 예측 임베딩"
→ 전체 곡 임베딩과 비교 → Top-K 추천

장점:
  1. 피처 엔지니어링 불필요 (same_artist_as_prev 등을 자동 학습)
  2. 장거리 패턴 포착 (3주 전 들었던 곡의 정보도 직접 참조)
  3. 병렬 처리로 학습 속도 빠름
```
