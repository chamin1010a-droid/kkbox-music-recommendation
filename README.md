# KKBox 음악 추천 대회 (Kaggle)

## 동기

이 대회는 [음원 생애주기 추천 알고리즘](https://github.com/chamin1010a-droid/music-recommender-lifecycle-based)의 **초기 버전**에서 출발했다. 당시 나는 사람이 직접 설계한 행동 피처(완주·스킵 패턴, 재생 빈도 추세 등)로 개인 추천기를 만들고 있었지만, 내 데이터는 1인 규모라 **이 피처 엔지니어링이 대규모에서도 통하는지** 가늠할 표본이 없었다. 그래서 실제 대규모 청취 데이터(유저 3만·곡 35만)가 있는 [KKBox Music Recommendation Challenge](https://www.kaggle.com/c/kkbox-music-recommendation-challenge)에 참가했다. 묻고 싶었던 건 두 가지다:

1. 사람이 손으로 만든 피처가 대규모 데이터에서도 통하는가?
2. 사람이 만든 피처 vs 모델이 스스로 학습하는 피처, 어디까지 갈 수 있는가?

> **이 저장소의 성격.** 처음엔 막연히 "내 추천 접근이 쓸모있는지 검증하는 자리"로 생각했지만, 그건 정확하지 않다. 이 대회의 과제는 *"유저가 곡을 재청취할 확률"*(이진 분류, AUC) 예측이고, 이는 추천기의 목표(*좋은 곡을 골라주기*)와 다른 문제다. 그래서 이건 추천기 검증이 아니라 위 두 질문에 답하는 **모델링 실험 기록**이다. (왜 둘이 다른 문제인지는 〈느낀 점〉에서 다룬다.)

| 항목 | 내용 |

| **대회** | [KKBox Music Recommendation Challenge](https://www.kaggle.com/c/kkbox-music-recommendation-challenge) |

## 실험 과정: 시행착오와 개선

### v3 — 시작: 기본 피처만으로는 부족하다

```
사용 피처: source_system_tab, source_screen_name, source_type, song_length, language, genre
결과:      Kaggle AUC 0.60
```

KKBox가 제공하는 메타데이터만으로는 한계가 있었다. "어떤 유저가 어떤 곡을 좋아하는가"에 대한 정보가 피처에 없었기 때문이다.

---

### v4 — Target Encoding 도입, 그리고 데이터 누수

유저별/곡별 재청취율을 피처로 추가하면 되지 않을까?

```
추가 피처: user_target_mean (유저별 재청취율), song_target_mean (곡별 재청취율)
결과:      Kaggle AUC 0.62
```

Kaggle 점수가 올랐지만, local 검증에서는 0.96이라는 비현실적인 점수가 나왔다.

**원인 분석**: Target Encoding 시 각 행의 target 값이 해당 행의 인코딩에 포함되고 있었다. 즉, 모델이 "정답을 보고 정답을 맞추는" 상황이었다. Local 검증에서는 이 누수가 드러나지 않았고, 실전 제출에서야 비로소 문제가 확인되었다.

**교훈**: 좋아 보이는 점수가 항상 진짜는 아니다. 특히 Target Encoding은 구현 방식에 따라 쉽게 누수가 발생한다.

---

### v5 — 누수 해결: Out-of-Fold Target Encoding

```
수정: 5-Fold 분할, 각 행의 target encoding은 해당 행이 포함되지 않은 fold의 통계로만 계산
추가: Smoothing 적용 — 출현 횟수가 적은 유저/곡은 전체 평균으로 수렴
결과: Kaggle AUC 0.65
```

local 점수가 크게 하락했지만, 이것이 진짜 실력이다. Kaggle에서도 안정적인 점수가 나왔고, 이후 모든 버전에서 이 방식을 유지했다.

---

### v7 — 시퀀스 피처: 행과 행 사이의 관계

```
추가 피처: 직전 곡과 같은 아티스트인가, 같은 장르인가, 세션 내 위치
결과:      Kaggle AUC 0.65 (유의미한 개선 없음)
```

트리 모델(LightGBM)은 각 행을 독립적으로 판단하기 때문에, "이전 곡과의 관계"를 피처로 만들어 넣어도 효과가 제한적이었다. 시퀀스 정보를 자연스럽게 활용하려면 RNN이나 Transformer 같은 시퀀스 모델이 필요하다는 것을 체감했다.

---

### v8 — SVD 임베딩: 유저-곡 잠재 벡터

```
추가: 유저-곡 상호작용 행렬 → TruncatedSVD → 32차원 잠재 벡터
      유저 벡터 32개 + 곡 벡터 32개 = 64개 피처 추가
결과: Kaggle AUC 0.70
```

이 버전에서 처음으로 유의미한 개선이 나왔다. SVD는 "이 유저와 이 곡이 잠재 공간에서 얼마나 가까운가"를 모델에게 알려주는 역할을 했다. 사람이 직접 설계할 수 없는 피처를 행렬 분해가 자동으로 만들어낸 셈이다.

---

### v9 — SVD 변형 실험

```
시도: SVD 차원 증가(64차원), 가중치 방식 변경(log-count) 등 실험
결과: Kaggle AUC 0.71
```

SVD 차원을 늘리거나 가중치를 바꿔도 극적인 개선은 없었다. 행렬 분해 방식으로 얻을 수 있는 정보량에 한계가 있다는 것을 확인했고, 별도로 Song2Vec(Word2Vec 기반 곡 임베딩)도 실험했으나 SVD와 중복되는 정보가 많아 유의미한 차이는 없었다.

---

## Phase 2: 딥러닝 도입

v9까지의 결론이 "모델이 스스로 발견할 수 없는 정보를 만들어주는 것이 피처 엔지니어링의 가치"였다면, 딥러닝은 그 정보 발견 자체를 자동화할 수 있지 않을까? 이 질문에 답하기 위해 NCF와 SASRec를 실험했다.

### NCF v1 — 딥러닝 단독: ID만으로 예측

```
모델: NeuMF (유저 임베딩 32d + 곡 임베딩 32d → MLP → 재청취 확률)
입력: 유저 ID, 곡 ID만 사용 (메타데이터 없음)
학습: 100만 행 샘플
결과: Kaggle AUC 0.61
```

SVD와 원리가 비슷하다. 둘 다 유저와 곡을 벡터로 표현한 뒤 궁합을 계산한다. 차이는 SVD가 내적(선형)으로 궁합을 계산하는 반면, NCF는 MLP(비선형)로 계산한다는 것. 하지만 성능은 LightGBM(0.71)에 한참 못 미쳤다.

**원인: Cold Start.** test 데이터의 약 19%는 train에 없는 유저 또는 곡이다. NCF는 ID 임베딩에 의존하므로 처음 보는 ID에 대해서는 사실상 랜덤 예측을 한다. 반면 LightGBM은 장르, 아티스트, 곡 길이 같은 메타데이터를 보기 때문에 처음 보는 곡이라도 유사한 특성의 곡에서 배운 패턴을 적용할 수 있다.

---

### NCF v2 — Side Feature 추가로 Cold Start 완화

```
추가: 장르, 언어, 아티스트, 나이, 성별, 도시 등 메타데이터를 모델에 함께 입력
학습: 전체 730만 행
결과: Kaggle AUC 0.67
```

처음 보는 유저/곡이라도 "30대 남성이 발라드 장르를 들었다"는 정보가 있으면 예측이 가능해졌다. v1(0.61) 대비 개선되었지만 여전히 LightGBM(0.71)보다 낮았다. 정형 데이터에서 딥러닝 단독으로 트리 모델을 이기기 어렵다는 것을 체감했다.

---

### NCF v3 — 하이브리드: 딥러닝 임베딩 + LightGBM

```
전략: NCF로 유저/곡 임베딩을 학습한 뒤, 그 벡터를 LightGBM의 피처로 추가
피처: 기존 v8 피처 + NCF 임베딩 65개 + SVD 임베딩 65개
결과: Kaggle AUC ~0.71 (개선 거의 없음)
```

"딥러닝이 잘하는 것(표현 학습)만 딥러닝에게 맡기자"는 전략이었다. 하지만 NCF 임베딩과 SVD 임베딩이 담고 있는 정보가 거의 같았다. 둘 다 "유저-곡 상호작용 행렬"에서 뽑은 것이니 당연한 결과였다. 피처 중요도를 보면 svd_dot과 ncf_dot이 1위, 2위를 차지하며 서로 중복되고 있었다.

**교훈: 같은 정보를 다른 방법으로 두 번 뽑는 것은 의미가 없다.**

---

### SASRec v4 — Transformer로 시퀀스 정보 추출 시도

```
모델: Self-Attentive Sequential Recommendation (Transformer 기반)
입력: 유저의 최근 50곡 청취 시퀀스 → Transformer → 유저 상태 벡터 (32d)
전략: 시퀀스 벡터를 LightGBM 피처로 추가 (NCF 대신 "새로운 정보")
결과: Kaggle AUC 0.64 (오히려 하락)
```

NCF와 SVD가 같은 정보를 뽑으니, 아예 새로운 종류의 정보(곡 순서 = 시퀀스)를 뽑아야 한다는 판단이었다. 하지만 KKBox 데이터의 근본적 한계에 부딪혔다:

1. **시퀀스 신뢰성 부재**: train.csv의 행 순서가 실제 시간순인지 보장할 수 없다. 타임스탬프가 없기 때문에 "연달아 들었다"와 "3일 뒤에 들었다"를 구분할 수 없다.
2. **시간 편향**: 최근 곡일수록 아직 재청취할 시간이 부족하여 target=0일 가능성이 높다. Transformer가 이 편향을 학습하면 일반화에 실패한다.
3. **노이즈 주입**: 신뢰할 수 없는 시퀀스에서 뽑은 임베딩 64개가 LightGBM에 노이즈로 작용하여, 기존 피처의 효과까지 희석시켰다.

## 느낀 점

### LightGBM 파트 (Phase 1)

**데이터 누수는 실전에서만 드러난다.** local 검증에서 0.96이 나왔을 때 "이상하다"고 느끼는 감각이 중요했다. 이 경험 이후 모든 프로젝트에서 "이 점수가 진짜인가?"를 먼저 의심하게 되었다.

**트리 모델은 분류를 자동으로 해준다.** 이전 음악 프로젝트에서 "이 곡은 Zone 2(듣다보니 좋아짐)인가, Zone 3(질린 곡)인가"를 사람이 직접 기준을 만들어 분류했다. 그런데 LightGBM은 이런 분류를 스스로 해버린다. 트리의 분기(split) 자체가 곧 분류이기 때문에, "스킵률 30% 이상이면 AND 재생 횟수 10회 이하이면" 같은 규칙을 사람이 피처로 만들어줄 필요가 없다. 트리가 알아서 그 조합을 찾아낸다.

**그러면 사람이 만들어줘야 하는 피처는 무엇인가.** 트리 모델이 스스로 할 수 없는 것 — 즉, "행과 행 사이의 관계"를 계산한 피처가 진짜 차별점이었다. v3~v7까지 만든 AND 조건 피처(장르가 같으면서 재생 횟수가 높은 경우 등)로는 Kaggle 점수가 움직이지 않았다. 반면 SVD는 "이 유저의 전체 청취 패턴"과 "이 곡의 전체 청취 패턴"을 행렬 분해로 요약한 것이고, 이것은 개별 행만 보는 트리가 절대 스스로 만들 수 없는 정보여서 0.65 → 0.70으로 올랐다.

### 딥러닝 파트 (Phase 2)

**정형 데이터에서 딥러닝 단독은 트리 모델을 이기기 어렵다.** NCF v2(0.67)는 Side Feature를 추가해도 LightGBM(0.71)에 못 미쳤다. 딥러닝이 강한 영역은 이미지, 텍스트, 시퀀스 같은 비정형 데이터다. 테이블 형태의 정형 데이터에서는 LightGBM/XGBoost 같은 트리 모델이 여전히 우세하다.

**같은 정보를 다른 방법으로 뽑아도 의미가 없다.** NCF 임베딩과 SVD 임베딩은 둘 다 "유저-곡 상호작용 행렬"에서 추출한 것이다. 방법이 다를 뿐(MLP vs 행렬분해) 원본 정보가 같으므로, 하이브리드로 합쳐도 성능 향상이 없었다.

**데이터 품질이 모델 복잡도보다 중요하다.** SASRec(Transformer)는 이론적으로 시퀀스를 잘 이해할 수 있지만, KKBox 데이터에는 타임스탬프도 반복 재생 정보도 없어서 의미 있는 시퀀스를 구성할 수 없었다. Transformer라는 강력한 도구가 있어도, 넣어줄 데이터의 품질이 낮으면 노이즈만 학습한다. 딥러닝 모델의 성능은 모델 구조가 아니라 데이터의 풍부함에 의해 결정된다.

### 돌아보며 — 음악 프로젝트와의 두 연결

**"학습된 표현 > 수작업 피처"는 한 번 더 반복됐다.** 이 대회에서 SVD(유저·곡 잠재벡터)가 수작업 AND 피처를 앞선 것과 똑같이, 이후 [음악 프로젝트](https://github.com/chamin1010a-droid/music-recommender-lifecycle-based)에서는 사람이 고른 음향 특성(librosa)을 음악으로 자기지도학습된 임베딩(MERT)이 앞질렀다. 서로 다른 두 도메인에서 독립적으로 같은 결론에 도달했다.

**재청취 예측은 '좋은 추천'의 잣대가 아니다 — 뒤늦게 안 것.** 이 대회의 타깃은 "유저가 그 곡을 다시 들을까"다. 그런데 사람은 듣던 걸 또 듣는 관성이 세서, 추천 품질을 재청취/미래 재생 예측으로 채점하면 '듣던 거 그대로 추천'이 늘 1등을 한다(repeat-baseline 함정). 즉 높은 AUC가 곧 좋은 추천을 뜻하지 않는다. 이 대회에선 이걸 의식하지 못했는데, 이후 음악 프로젝트를 발전시키며 *서빙(무엇을 골라줄까)*과 *평가(추천이 좋았나 채점)*가 다른 문제임을 분명히 갈랐다. kkbox는 이 자각의 출발점이었던 셈이다.

## 프로젝트 구조

```
kkbox/
├── eda.py                     # 탐색적 데이터 분석 (시각화 10종)
├── pipeline.py                # v1 기본 파이프라인
├── pipeline_v3.py             # v3 Source 피처
├── pipeline_v4.py             # v4 Target Encoding (누수 버전)
├── pipeline_v5.py             # v5 OOF Target Encoding (해결)
├── pipeline_v7.py             # v7 시퀀스 피처
├── pipeline_v8.py             # v8 SVD 임베딩
├── pipeline_v9_experiment.py  # v9 SVD 변형 실험
├── ncf_v1.py                  # NCF 단독 (ID만 사용)
├── ncf_v2.py                  # NCF + Side Feature
├── ncf_v3_hybrid.py           # NCF 임베딩 + LightGBM 하이브리드
├── sasrec_v4.py               # SASRec(Transformer) + LightGBM
├── sasrec_model.pt            # 학습된 SASRec 가중치 (53MB, 저장소 미포함 — 재현 시 생성)
├── song2vec_study.py          # Song2Vec 실험
├── deep_learning_notes.md     # 딥러닝 자작 학습 노트 (로지스틱 회귀→Transformer)
└── eda_output/                # EDA 시각화 결과 (PNG)
```

## 기술 스택

- **트리 모델**: LightGBM (gbdt, 2000 trees, learning_rate=0.1)
- **딥러닝**: PyTorch (NCF, SASRec/Transformer)
- **검증**: 5-Fold Stratified CV + Kaggle Public LB
- **피처**: OOF Target Encoding, TruncatedSVD (32차원), NCF 임베딩, SASRec 시퀀스 임베딩
- **라이브러리**: pandas, numpy, scikit-learn, lightgbm, scipy, torch
